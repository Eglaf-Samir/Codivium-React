Section 1

Introduction

The Codivium Performance Dashboard is a self-contained analytics front-end that visualises a learner's coding and MCQ practice progress using the KaTeX mastery scoring model. It is designed to be embedded into any web application as a standalone page or iframe, receiving all data from a single JSON payload produced by the backend.

This guide covers the complete integration path: what data to pull from your database, how to pass it through the scoring and aggregation pipeline, how to assemble the final JSON payload, how to deliver that payload to the dashboard, and how to handle the action calls that the dashboard's CTA buttons generate.

Reading this guide

Sections 1–5 give context. Sections 6–13 are the core integration path and should be read by every developer implementing the integration. Sections 14–16 cover CTA handling, which requires additional application-side work. Sections 17–21 cover supporting concerns.

Section 2

Purpose of this document

This document is the definitive technical reference for integrating the Codivium dashboard into a host application. It is intended for:

  • Backend engineers who need to understand what data the dashboard requires and how to supply it.
  • Full-stack developers wiring the scoring pipeline into an existing application backend.
  • Platform architects deciding where the dashboard fits in the overall system and what API contracts it needs.

It does not cover the internal mathematics of the scoring model (that is the KaTeX document) or the internal workings of the frontend JavaScript (that is the developer comments in the JS source files). It covers only the integration boundary between your application and the dashboard.

Section 3

Purpose of the dashboard

The dashboard serves three functions simultaneously:

  • Mastery reporting. It shows the learner their current Codivium Score — a 0–100 mastery metric that combines breadth of coverage and depth of practice across coding and MCQ tracks — and the key supporting metrics (breadth per track, depth per category, points, efficiency).
  • Momentum reporting. It shows points earned all-time and over the last 30 days, and an efficiency metric (pts/hr) that rewards fast, clean accepted solves. These are deliberately separate from the mastery score so the learner can distinguish "improving" from "practising a lot".
  • Prescription. Each panel contains a CTA button that, when clicked, launches a pre-configured practice session based on the analysis of that panel's data. The dashboard is designed to close the practice loop: Analytics → Recommendation → Session → New evidence → Updated analytics.

The dashboard does not store any data. It is a pure rendering layer. All computation happens in the backend before the payload is sent.

Section 4

Architecture and layers

The full system has seven layers from database to dashboard render:

Database Your application's DB — coding sessions, MCQ attempts, exercise catalogue, user accounts.
↓
Raw data CodiviumRawData — sessions normalised into the KaTeX input contract via CodiviumRawDataBuilder.
↓
Scoring CodiviumKatexScoring.ComputeAll() — produces breadth, depth, Codivium Score, points, efficiency from raw data.
↓
Aggregation Chart-specific aggregators — convergence heatmap, depth-by-category, allocation, time-on-platform, MCQ performance, heatmap focus rankings, insights, recommended actions.
↓
Payload DTOs DashboardPayloadV2Adapter.BuildFromRaw() — assembles all aggregated data into the canonical DashboardPayloadV2 JSON shape.
↓
API endpoint GET /api/dashboard/payload — serialises the DTO to JSON and returns it to the browser, optionally caching it.
↓
Dashboard JS CodiviumInsights.init() — receives the JSON, renders all charts and panels, wires up CTA button click handlers.

The scoring engine and all aggregators are stateless pure functions. They take raw data and a timestamp in, and return structured results out. There are no side effects and no DB calls inside the scoring layer.

Section 6

Data required by the dashboard

Background: what Codivium is and how it works

Before describing the data fields, it helps to understand what Codivium is doing and what kind of activity it is measuring — because the data structure flows directly from this.

Codivium is a deliberate practice platform with three distinct tracks of activity:

  • Micro Challenges — short, focused algorithmic coding exercises. A learner is given a problem and writes code to solve it. The code is tested against a suite of automated unit tests. The learner can submit as many times as they like. A session is "accepted" when all unit tests pass. Micro challenges are designed to be completable in 15–45 minutes and test one narrow skill at a time (e.g. tree traversal, binary search, sliding window).
  • Interview Preparation — longer, more complex coding exercises that simulate real technical interview problems. These test the same underlying skill categories as micro challenges but at greater difficulty and breadth. A session is accepted when all unit tests pass, by the same mechanism.
  • MCQ (Multiple Choice Questions) — knowledge-check quizzes that test conceptual understanding of the same skill categories. A quiz is a batch of questions on a single category and difficulty level. Unlike coding tracks, MCQ has no pass/fail notion — every quiz attempt counts, and the score is a percentage of questions answered correctly.

The dashboard does not care about most of what happens inside a session. It only needs to know: what was attempted, what was the outcome, when did it happen, how many tries did it take, and how long did it take. Everything else — the Codivium Score, breadth, depth, points, efficiency, the heatmap, the insights — is computed from just those facts.

The four data inputs

The scoring engine requires exactly four inputs from your database. You do not pre-compute any scores — you supply the raw activity data and the engine produces all scores from scratch on every call.

Important principle

Supply all historical sessions for the user, not just recent ones. The scoring engine applies its own recency decay (work from 30 days ago counts less than work from today). If you filter to only recent sessions at the DB layer, the engine will compute incorrect scores because it will be missing the full evidence history.

Input 1 — Coding sessions

A coding session represents one exercise attempt that was ultimately accepted — meaning the learner submitted code that passed all unit tests for that exercise. Each row in this dataset corresponds to one complete exercise attempt, from the moment the learner started until the final passing submission.

If a learner solved the same exercise three times on different days, that produces three session rows (one per completion), all with the same ExerciseId but different timestamps and potentially different attempt counts.

FieldRequired?TypeFull explanation
SessionId Required string The unique identifier for this specific session. A session is one instance of a learner working on one exercise. If the same learner completes the same exercise on Tuesday and again on Saturday, those are two different sessions and must have two different SessionIds. This field is used solely to link the session row to its attempt rows (see Input 2 below). It never appears in any score computation. Use your database's primary key for the session table — a UUID, an auto-increment integer, or any stable unique string all work.
ExerciseId Required string The unique identifier for the exercise (problem) itself, not for this specific completion of it. An exercise is a single coding problem — for example, "Reverse a linked list" or "Find the longest common subsequence". An exercise has a fixed definition (problem statement, test suite, category, difficulty) that never changes. Every time a learner completes this exercise, the session row for that completion carries the same ExerciseId.

The scoring engine uses ExerciseId for two critical purposes:
  • Distinct-solve counting. Points are only awarded once per exercise (the first time a learner accepts it). The engine identifies the "first solve" by finding the earliest session for each ExerciseId. Without a stable ExerciseId this is impossible.
  • Redo spacing. After the first solve, the learner can earn "redo points" by solving the same exercise again — but only if at least 7 days have passed since the previous solve. The engine checks this spacing using the ExerciseId to group sessions for the same exercise. Without a stable ExerciseId, redo points cannot be computed.
Critical: ExerciseId must be globally unique across all tracks. If your Micro Challenges exercises and your Interview Preparation exercises are stored in separate tables and their IDs are independent sequences (e.g. both have an exercise with ID "42"), then exercises with the same number but in different tracks would be incorrectly treated as the same exercise. The safest way to avoid this is to prepend the track identifier to the exercise ID when constructing the ExerciseId for the scoring engine:

micro_42 for micro challenge exercise #42
interview_42 for interview preparation exercise #42
mcq_42 for MCQ exercise #42 (if applicable)

If your exercise IDs are already globally unique (e.g. GUIDs from a single exercises table), this prefixing is not needed. If in doubt, add the prefix — there is no downside to a slightly longer ID string.
TrackId Required string Which of the two coding tracks this session belongs to. Must be exactly "micro" or "interview" (case-insensitive; leading/trailing whitespace is stripped automatically).

The track determines:
  • Which category universe the session's category is checked against (micro and interview each have their own list of available categories in the config).
  • Which track's breadth and depth scores the session contributes to (micro sessions contribute to micro breadth and micro depth; interview sessions to interview breadth and interview depth).
  • The activity weight applied in the evidence formula (w_a — currently 1.0 for both tracks).
If your database uses different names (e.g. "track_type = 1" for micro and "track_type = 2" for interview), map them to the canonical strings in your DB query or in the builder.
CategoryId Required string The skill category that this exercise belongs to. A category is a named area of computer science knowledge and problem-solving skill. Examples: "Arrays", "Trees", "Dynamic Programming", "Graphs", "Sorting".

Categories are the fundamental unit of breadth measurement. The scoring engine counts how many categories a learner has meaningful evidence in, how evenly evidence is distributed across categories, and how much evidence is in each category for depth computation. Without correct categories, breadth and depth scores will be meaningless.

CategoryId must exactly match one of the category names in your scoring-config.v1.json file (under categories.available_by_track.micro or categories.available_by_track.interview). The match is case-insensitive and whitespace is trimmed, but the name itself must be one of the configured values. Sessions with a CategoryId not in the config are silently ignored (they contribute no evidence). If you find that certain categories always contribute zero evidence, check for a spelling mismatch between your DB values and the config.

The config file's category list is also used as the denominator in breadth calculations — if you have 14 categories configured and the learner has evidence in 7 of them, their coverage is 50%. If you add new exercises in a new category, you must also add that category to the config before the scoring engine will recognise it.
Difficulty Required string The difficulty level of the exercise. Must be one of three values: "basic", "intermediate", or "advanced".

Difficulty affects the scoring in several ways:
  • Evidence weighting (f_d). Harder exercises contribute more evidence per accepted solve: basic = 1.0, intermediate = 1.3, advanced = 1.6. This means an advanced accept is worth 60% more evidence than a basic accept, all else being equal.
  • Points. First-solve and redo points are also multiplied by f_d, so harder exercises generate more points.
  • Time penalty constants. The quality factor that penalises slow solves uses different time thresholds per difficulty: basic 90 minutes, intermediate 120 minutes, advanced 160 minutes. A basic solve taking 45 minutes is considered fast; an advanced solve taking 90 minutes is also considered fast for its difficulty level.
Legacy integer values 1, 2, 3 are automatically converted to basic, intermediate, advanced respectively. The strings "easy", "medium", "hard" are also accepted as synonyms. All other values fall back to "basic" with a warning.
CompletedAtUtc Required datetime (UTC) The UTC timestamp of when the learner's final passing submission was made — the moment the exercise was accepted. This is not the session start time or the time the learner first opened the problem. It is specifically when they submitted code that passed all unit tests for the final time.

This timestamp is used in two places:
  • Recency decay. The scoring engine computes how many days ago each accepted solve occurred and applies a decay factor: decay = 0.5^(days/15). This means work done 15 days ago counts half as much as work done today. Work done 30 days ago counts a quarter as much. This keeps the mastery score reflecting the learner's current ability, not just their entire career history. Without CompletedAtUtc the recency decay cannot be applied and the engine will throw an error.
  • 30-day rolling window. The Points (30 days) metric counts first-solves completed within the last 30 days. This requires knowing exactly when each solve occurred.
Store this as a proper UTC datetime in your database, not as a local time. If your system stores local times, convert to UTC before passing to the engine.
UnitTestsTotal Required integer The total number of automated unit tests that must pass for this exercise to be considered accepted. In Codivium, a coding exercise is considered solved only when all unit tests pass. The engine determines whether a submission attempt is the "accepted" attempt by checking whether UnitTestsPassed >= UnitTestsTotal. If UnitTestsTotal is 10, then a submission that passes 8 tests is not accepted; one that passes 10 is accepted.

This value is typically fixed per exercise (the same exercise always has the same test suite). If your DB stores this on the exercise definition table rather than per session, join it in. If different exercises have different numbers of tests (which they normally will — a complex tree problem might have 15 tests while a simple array problem has 8), this value must reflect the actual count for each exercise.

This field does not affect scoring computations directly — it is only used to determine acceptance. Once a session is classified as accepted, UnitTestsTotal plays no further role.
StartedAtUtc Optional datetime (UTC) The UTC timestamp of when the learner opened the exercise and began their session. This is not used in any score computation and does not affect any dashboard metric. It is included in the raw data contract for telemetry and debugging purposes — for example, if you want to verify that time-to-submission figures are plausible relative to the session start time. If you do not record session start times, leave this null.

Input 2 — Submission attempts

Within a single coding session, the learner may submit their code multiple times before it passes all tests. Each of those individual submissions is a "submission attempt". This table has one row per submission, and every session row from Input 1 must have at least one attempt row here.

For example: a learner opens "Reverse a Linked List", submits code that passes 6 out of 10 tests (attempt 1), revises and submits again passing 9 out of 10 (attempt 2), revises again and passes all 10 (attempt 3). That session has 3 attempt rows. The session is accepted because attempt 3 passed all tests.

The scoring engine uses attempts to compute quality factors that affect evidence and efficiency. Specifically: how many attempts were needed before acceptance (fewer is better — this is the convergence quality factor), and how many minutes elapsed before the accepted attempt (faster is better — this is the time quality factor).

FieldRequired?TypeFull explanation
SessionId Required string The identifier of the session this attempt belongs to. This must exactly match a SessionId from Input 1. The engine uses this to group attempts under their parent session. Attempts with no matching session will be silently ignored, and sessions with no matching attempts will be excluded from evidence computation entirely (because acceptance cannot be determined without attempt data).
UnitTestsPassed Required integer The number of unit tests that the learner's code passed on this specific submission. This is compared against the session's UnitTestsTotal to determine whether the attempt is an accepted solve. An attempt where UnitTestsPassed equals UnitTestsTotal is accepted. An attempt where UnitTestsPassed is less than UnitTestsTotal is not accepted (it is a failed attempt — the learner needs to try again).

The engine scans all attempts in a session and identifies the last one where UnitTestsPassed >= UnitTestsTotal. "Last" here means the attempt with the highest TimeToSubmissionMinutes value (i.e. the final passing submission). The number of attempts up to and including that final passing submission is the "attemptsToAccepted" count. This count feeds the convergence quality factor (q_conv) which reduces evidence and points for sessions requiring many attempts.

If your system awards partial credit (e.g. partial pass is meaningful for scoring), use the raw count of tests passed. The engine correctly handles partial passes in the attempt sequence — only the final attempt where all tests pass determines acceptance.
TimeToSubmissionMinutes Required decimal The cumulative number of minutes elapsed from the start of the session to the moment this specific submission was made. This is measured from session start, not from the previous attempt. So if a learner starts at 10:00, submits at 10:12 (attempt 1), and submits again at 10:28 (attempt 2), the values are 12.0 and 28.0 — not 12.0 and 16.0. The values must be monotonically non-decreasing within a session (each attempt's time must be greater than or equal to the previous one).

This field is used to derive TimeToACMinutes — the time-to-accepted-solution. If your DB stores an explicit TimeToACMinutes on the session row, use that directly and this field is only used as a fallback. If not, the engine takes the TimeToSubmissionMinutes of the last accepted attempt as the time-to-AC.

TimeToACMinutes drives two scoring factors:
  • Time quality factor (q_time). Faster solves earn slightly more evidence. The effect is mild and clamped: the slowest possible solve still earns 60% of the quality weight of the fastest.
  • Efficiency (pts/hr). Time is the denominator in the efficiency formula. More points earned per minute of practice = higher efficiency. This is the main place where time directly affects a displayed metric.
If your system records submission timestamps but not cumulative elapsed minutes, compute the cumulative time by subtracting the session start time from each submission timestamp and converting to minutes.
What if a learner passes, then submits again and fails?

This can happen — some learners refactor after passing and inadvertently break tests. The engine handles this correctly: it scans all attempts and keeps the last accepted attempt (not the first). So if attempts are [fail, pass, fail, pass], attemptsToAccepted = 4 (all four attempts are counted as part of the work), and the time used is the time of attempt 4. The session is still considered accepted because the last passing submission is what matters.

Input 3 — MCQ quiz attempts

An MCQ quiz attempt is one completed quiz. A quiz is a set of multiple-choice questions on a specific skill category at a specific difficulty level. Unlike coding sessions, there is no pass/fail for MCQ — every completed quiz attempt contributes evidence. A score of 30% still contributes evidence; it just contributes less than a score of 90%.

The scoring engine treats each MCQ quiz attempt as a separate event with its own score, timestamp, category, and difficulty. There is no concept of "sessions" or "attempts within a session" for MCQ — each quiz attempt is self-contained.

MCQ contributes to two things in the scoring model: MCQ breadth (how broadly the learner has tested their knowledge across MCQ categories, weighted by recency and score quality), and all-time MCQ points (a career progression signal based on the total count of distinct questions ever answered correctly).

FieldRequired?TypeFull explanation
QuizId Required string A stable, unique identifier for this specific quiz attempt. A "quiz attempt" is one instance of a learner completing a quiz. If the same learner takes the same "Arrays Basic" quiz three times on different days, each completion is a separate quiz attempt with a different QuizId.

This is used to identify the quiz attempt uniquely — for example, to look up its answers when computing the lifetime distinct-correct-question count (see Input 4). Use your database's primary key for the quiz attempt table.
TrackId Required string The track this quiz belongs to. Must be "mcq" (case-insensitive). All MCQ quizzes belong to the MCQ track. This field tells the engine to route this attempt to the MCQ evidence accumulator rather than the coding evidence accumulator. If your system groups MCQ quizzes by sub-type (e.g. "mcq_basic", "mcq_advanced"), you can use any string that starts with "mcq" — it will be normalised to the "mcq" track.
CategoryId Required string The skill category this quiz tests. Uses the same category taxonomy as coding exercises. A "Trees MCQ" quiz has CategoryId = "Trees". This must match one of the categories in categories.available_by_track.mcq in your scoring config. Each quiz tests exactly one category.

MCQ breadth is computed per category: does the learner have meaningful quiz evidence in this category? A learner who has only taken Arrays quizzes will have MCQ breadth coverage of 1/14 (if there are 14 MCQ categories). A learner who has taken quizzes across 10 of the 14 categories will have much higher coverage.
Difficulty Required string The difficulty level of the quiz. Same three values as coding: "basic", "intermediate", "advanced". Each quiz has a single difficulty level (a quiz is not a mix of difficulties). The difficulty affects the evidence contribution via the same difficulty factor f_d used for coding: advanced MCQ quizzes contribute 1.6× as much evidence as basic ones, weighted by the score percentage. A high score on an advanced quiz earns substantially more MCQ breadth evidence than the same score on a basic quiz.
ScorePercent Required decimal 0–100 The percentage score the learner achieved on this quiz — i.e. the fraction of questions answered correctly, expressed as a number between 0 and 100. If the quiz had 20 questions and the learner answered 14 correctly, ScorePercent = 70.0.

The score is used in the MCQ evidence formula: evidence = (ScorePercent/100)^1.5 × difficulty_factor × recency_decay. The exponent 1.5 means higher scores contribute disproportionately more evidence than lower scores. A 90% score contributes 85% of the maximum possible evidence for that quiz; a 50% score contributes only 35%. A 0% score contributes no evidence at all. This incentivises genuine understanding rather than low-effort quiz completion.

The score may reflect partial credit if your quiz system supports it (e.g. 7.5 out of 10 → 75.0). Use the raw percentage; do not round to whole numbers unless your system only tracks whole-number percentages.
CompletedAtUtc Required datetime (UTC) The UTC timestamp of when the learner submitted the quiz. Used for the same recency decay applied to coding sessions: quiz evidence from 15 days ago counts half as much as quiz evidence from today. This keeps MCQ breadth reflecting the learner's current knowledge state rather than historical knowledge they may have forgotten.

All the same rules as coding CompletedAtUtc apply: must be UTC, must be a real timestamp (not null), and should not be in the future.
TotalQuestions Optional integer The total number of questions in the quiz. Not used in any score computation. Displayed in the MCQ performance chart tooltips so the learner can see "14/20 correct" rather than just "70%". If not provided, tooltips will only show the percentage.
Correct Optional integer The number of questions the learner answered correctly on this quiz attempt. Not used in score computation (ScorePercent is used instead). Shown alongside TotalQuestions in chart tooltips (e.g. "14/20 correct"). If not provided, the tooltip will not show the raw correct count.
MinutesSpent Optional decimal The number of minutes the learner spent completing this quiz. Used in the time-on-platform daily series, which aggregates all practice time (coding + MCQ) into a per-day total. If not provided, the engine estimates 6 minutes per quiz attempt as a default. If your system records actual time-on-quiz (e.g. from start timestamp to submission timestamp), use that. Accurate time data makes the time-on-platform chart more meaningful.

Input 4 — Lifetime unique correct MCQ question count

This is a single integer value, not a table. It represents the total count of distinct MCQ question IDs that this specific learner has ever answered correctly across their entire history on the platform — not just in the current query window.

This value is used for one specific metric: MCQ all-time points. The formula is Points_mcq = 0.8 × √(UniqueCorrectMcqAllTime). The square root is intentional — it gives early MCQ progress significant reward (going from 0 to 100 distinct correct questions is very rewarding) while preventing the metric from growing unboundedly for users with very long histories.

This count cannot be derived from the quiz-level data in Input 3, because that data operates at the quiz-attempt level. To know how many distinct questions have been answered correctly, you need per-question-answer data: a record of which specific questions were in each quiz and which ones were answered correctly. The formula is:

SELECT COUNT(DISTINCT qa.question_id)
FROM mcq_question_answers qa
WHERE qa.user_id = @userId
  AND qa.is_correct = true;

If your platform does not yet record per-question answers, pass 0 for this value. The MCQ points component will be 0, but all other scoring (MCQ breadth, coding breadth and depth, coding points, efficiency) will work normally. You can add per-question tracking later and the metric will begin computing correctly from that point.

Do not pass -1

The default value of this field in the data contract is -1, which signals "not provided". If you pass -1 and no legacy alias is present, the scoring engine will throw a runtime exception. Always pass 0 if you have no per-question data, never -1.

Section 7

Data structure — the payload

After the scoring engine processes the raw data (Section 6), all results are assembled into a single JSON object called the dashboard payload. This payload is what the frontend receives and what drives every number, chart, and piece of text on the dashboard. You never construct this object by hand — the adapter function DashboardPayloadV2Adapter.BuildFromRaw() builds it for you from the raw data. This section describes the structure of the resulting JSON so you can understand and debug it.

Top-level shape

The payload has six top-level keys:

{
  "version": "dashboard-payload-v2",        // contract version identifier
  "meta": { /* configuration and settings */ },
  "overall": { /* all headline metrics + global data */ },
  "combinedCoding": { /* chart data for micro + interview merged */ },
  "micro": { /* chart data for micro challenges only */ },
  "interview": { /* chart data for interview preparation only */ },
  "mcq": { /* MCQ metrics and performance chart data */ }
}

The meta block

The meta block carries configuration values that the frontend needs to know but that are not data values. The most important field is sessionStartEndpoint — the URL the dashboard's CTA buttons will POST to when clicked. This is set by you when you call BuildFromRaw().

The overall block

This is the largest block. It contains everything that applies to the learner globally (not broken down by track). It has four sub-keys:

  • metrics — all the numeric KPI values shown on the dashboard (see table below)
  • timeOnPlatform — a daily series of practice minutes (combined coding + MCQ)
  • insights — a dictionary of pre-computed analysis text objects, one per metric key and panel. These are what appear in the info pane when the learner clicks an (i) button.
  • recommendedActions — an array of CTA recommendation objects, one per dashboard panel, that drive the action buttons

The combinedCoding, micro, and interview blocks

These three blocks all share the same internal structure. They contain: the depth-by-category bar chart data, the time-allocation-by-category chart data, and the convergence heatmap matrix. The difference is the scope: combinedCoding merges evidence from both coding tracks (most charts use this view), while micro and interview contain track-specific views that the dashboard uses when the learner switches to a track-specific layout.

The mcq block

Contains the MCQ breadth score, the weighted MCQ performance score, and the detailed MCQ performance chart data broken down by difficulty level (Basic / Intermediate / Advanced) and by category.

Key metrics fields — explained in plain language

These are the numeric values that appear as the headline chips across the top of the dashboard. Every value here is computed from the raw data you supplied — you never set these directly.

FieldRangeWhat it measures in plain language
codiviumScore 0–100 The single headline mastery number. It is the harmonic mean of overall breadth and overall depth. The harmonic mean (rather than the arithmetic mean) means that if either breadth or depth is weak, the score is pulled down significantly. A learner who has practised deeply in only two categories but ignored the rest will score lower than their depth alone would suggest, because their breadth is low. A learner who has touched every category lightly but never practised anything deeply will also score lower than their breadth suggests, because their depth is low. You need both to score well.
breadthOverall 0–100 How widely spread the learner's practice is across all three tracks. It is the weighted average of micro breadth (40%), interview breadth (40%), and MCQ breadth (20%). A score of 0 means no evidence in any track. A score of 100 would mean perfectly even evidence across all categories in all tracks — in practice, 60–80 is considered strong.
breadthMicro 0–100 How widely spread practice is specifically within the Micro Challenges track. Combines coverage (what fraction of micro categories have meaningful evidence), balance (how evenly the evidence is distributed — not all piled into one category), and confidence (a stabiliser that prevents early users with just one or two solves from scoring artificially high).
breadthInterview 0–100 Same as breadthMicro but for the Interview Preparation track.
breadthMcq 0–100 How widely spread MCQ practice is — how many MCQ categories the learner has taken meaningful quizzes in, how evenly spread their quiz activity is, and how much total quiz evidence they have accumulated.
depthOverall 0–100 How deeply the learner has practised within the categories they have worked on. Uses a saturating curve: the first few solves in a category raise depth quickly; adding more solves to a category already at high depth yields diminishing returns. This prevents "grinding one category forever" from being an effective strategy. Averages micro depth and interview depth with equal weight.
depthMicro 0–100 Evidence-weighted average depth across all micro challenge categories. Categories where the learner has accumulated more evidence contribute more to this average.
depthInterview 0–100 Same as depthMicro but for interview preparation exercises.
codiviumPointsAll Uncapped Career total points — an uncapped number that grows throughout the learner's time on the platform. It combines: first-solve points (earned once per exercise, weighted by difficulty and submission quality), redo points (earned by re-solving exercises at least 7 days after the previous solve, with logarithmic diminishing returns to prevent gaming), and MCQ points (a separate component proportional to the square root of lifetime distinct correct questions). Points are designed to grow even for learners who have already plateaued their mastery score — they reward sustained practice volume.
codiviumPoints30 Uncapped Points earned specifically in the last 30 days. This is the "momentum" metric — it shows whether the learner is currently active. A learner who had a productive history but has not practised recently will have high codiviumPointsAll but low codiviumPoints30.
efficiencyPtsPerHr Uncapped Quality-weighted points earned per hour of practice. Rewards producing correct solutions quickly. A learner who grinds through 5-attempt, 90-minute solves will have lower efficiency than one who solves cleanly in 2 attempts and 25 minutes. Recent sessions are weighted more heavily than old ones (same recency decay as the mastery score). This metric is most useful for identifying whether a learner's practice strategy is efficient or whether they are spending a lot of time on difficult solves without proportionate quality improvement.
firstTryPassRate 0–100% The percentage of accepted coding sessions where the learner passed all unit tests on their very first submission. A high first-try pass rate indicates strong problem-solving and planning skills — the learner thinks through their solution before coding rather than submitting to see what breaks. A low rate does not necessarily indicate weakness; on hard problems, multiple attempts are expected.
avgAttemptsToAC ≥1.0 The mean number of submissions across all accepted coding sessions. Lower is better in general, but context matters: 2.5 attempts average on advanced exercises is quite good; 2.5 on basic exercises suggests the learner is not planning solutions carefully before submitting.
medianTimeToACMinutes ≥0 The median (middle value, not average) time in minutes from session start to the final passing submission. Median is used instead of average because a few very long sessions would skew an average significantly — the median is more representative of typical performance. This feeds directly into the efficiency metric and gives the learner a sense of their typical solve speed.
weightedMcqScore 0–100 A summary quality score for MCQ performance, independent of how many quizzes have been taken. Computed as the difficulty-weighted mean quiz score: advanced quiz scores count 1.6× as much as basic quiz scores in the average. This gives a single "how well am I doing on MCQs" number that accounts for the fact that performing well on advanced content is worth more than performing well on basic content.
Section 8

Passing data into the payload

You never construct DashboardPayloadV2 directly. The correct integration path is:

  1. Query your database for the user's sessions, attempts, MCQ attempts, and U_all.
  2. Load your ScoringConfig (from scoring-config.v1.json or your config service).
  3. Pass all of the above to CodiviumRawDataBuilder.Build() to produce a CodiviumRawData object.
  4. Pass CodiviumRawData to DashboardPayloadV2Adapter.BuildFromRaw().
  5. Serialise the resulting DashboardPayloadV2 to JSON and return it from your API endpoint.

The adapter handles all scoring, all aggregation, all chart data, all insight text, and all CTA recommendation selection internally. You do not need to call any other functions.

Section 9

Database queries

The following pseudo-SQL illustrates the queries your DB layer needs to supply. Adapt field names to your schema.

Coding sessions

SELECT
    s.session_id       AS SessionId,
    s.exercise_id      AS ExerciseId,
    s.track_id         AS TrackId,          -- 'micro' | 'interview'
    s.category_id      AS CategoryId,
    s.difficulty       AS Difficulty,       -- 1/2/3 or 'basic'/'intermediate'/'advanced'
    s.unit_tests_total AS UnitTestsTotal,
    s.started_at_utc   AS StartedAtUtc,
    s.completed_at_utc AS CompletedAtUtc
FROM coding_sessions s
WHERE s.user_id = @userId
  AND s.completed_at_utc IS NOT NULL
ORDER BY s.completed_at_utc ASC;
Tip

Include all accepted sessions, not just recent ones. The KaTeX recency decay reduces the weight of old sessions automatically. Filtering to a date range at the DB layer will cause the scoring model to underestimate historical evidence.

Submission attempts

SELECT
    a.session_id              AS SessionId,
    a.unit_tests_passed       AS UnitTestsPassed,
    a.time_to_submission_mins AS TimeToSubmissionMinutes
FROM coding_attempts a
INNER JOIN coding_sessions s ON s.session_id = a.session_id
WHERE s.user_id = @userId
  AND s.completed_at_utc IS NOT NULL
ORDER BY a.session_id, a.time_to_submission_mins ASC;

MCQ attempts

SELECT
    q.quiz_attempt_id    AS QuizId,
    'mcq'                AS TrackId,
    q.category_id        AS CategoryId,
    q.difficulty         AS Difficulty,
    q.score_percent      AS ScorePercent,
    q.total_questions    AS TotalQuestions,
    q.questions_correct  AS Correct,
    q.minutes_spent      AS MinutesSpent,
    q.completed_at_utc   AS CompletedAtUtc
FROM mcq_quiz_attempts q
WHERE q.user_id = @userId
  AND q.completed_at_utc IS NOT NULL
ORDER BY q.completed_at_utc ASC;

Lifetime unique correct MCQ count

SELECT COUNT(DISTINCT qa.question_id)
FROM mcq_question_answers qa
WHERE qa.user_id = @userId
  AND qa.is_correct = 1;
If you do not have per-question tracking yet

Pass uniqueCorrectMcqAllTime: 0. MCQ points will be 0 but all other scoring will work normally.

Section 10

Scoring configuration

The scoring config has two jobs: it defines the available categories per track (the universe your breadth denominator uses), and it carries all the KaTeX mathematical constants.

Category lists

The categories.available_by_track section tells the scoring engine the complete set of categories that exist for each track. This is used as the denominator in breadth calculations and as the axes for all charts. You must keep this in sync with your exercise catalogue. If you add a new category to your catalogue, add it here too — otherwise breadth will be computed against the wrong denominator.

// From scoring-config.v1.json (abbreviated)
{
  "categories": {
    "available_by_track": {
      "micro":     ["Arrays", "Strings", "Trees", /* ... */],
      "interview": ["Arrays", "Strings", "Trees", /* ... */],
      "mcq":       ["Arrays", "Strings", "Trees", /* ... */]
    }
  }
}

Loading the config

var cfgJson = File.ReadAllText("config/scoring-config.v1.json");
var cfg = JsonSerializer.Deserialize<CodiviumScoring.ScoringConfig>(
    cfgJson,
    new JsonSerializerOptions { PropertyNameCaseInsensitive = true })
    ?? throw new InvalidOperationException("Failed to parse scoring config.");

CodiviumConfigValidation.ValidateOrThrow(cfg); // always validate before use
Section 11

Aggregation functions — what, how, and why

These are the individual aggregators that produce the chart and panel data. You do not call these directly in normal integration — BuildFromRaw() calls them all for you. This section is reference documentation so you can understand what each one does and troubleshoot if needed.

FunctionProducesWhy it exists
CodiviumKatexScoring.ComputeAll()All headline scores: Codivium Score, breadths, depths, points, efficiency, evidence by categoryThe core KaTeX mathematics. Single source of truth for all scalar metrics shown on the dashboard.
CodiviumDepthByCategory.ComputeDepthByCategory()Per-category depth values for the depth bar chartApplies the saturation curve to combined micro + interview evidence. Separate from the track-level depth in ComputeAll because it merges evidence across tracks by canonical category key.
CodiviumConvergenceHeatmap.BuildConvergenceHeatmap()A matrix: for each category and attempt bucket A1–A5, the mean unit-test pass ratioReveals how quickly code quality converges to correct across attempts. Shows not just whether sessions passed, but how good the code was at each attempt number.
CodiviumHeatmapFocusBuilder.BuildFocus()Five ranked category lists, one per focus mode (Impact, Worst Convergence, Most Time, Lowest Depth, Highest Friction)Powers the heatmap dropdown. Each mode surfaces a different actionable lens on where to focus practice next.
CodiviumAllocation.BuildAllocationByCategory()Total minutes and solved count per category (combined coding)Shows where practice time is going. Used in the allocation chart and as input to heatmap focus modes.
CodiviumTimeOnPlatform.BuildTimeOnPlatform()A daily series of minutes: coding time + MCQ time per calendar dayPowers the time-on-platform chart and is used in the insight text for the time panel.
CodiviumMcq.BuildMcqPayload()MCQ performance by difficulty and category, overall correct counts, evidence cellsPowers the three MCQ chart tabs (Basic / Intermediate / Advanced). The data structure matches the chart's axis definition.
CodiviumInsights.BuildAll()A dictionary of structured insight objects, one per metric key and panelPre-computes all the analysis text that appears in the info pane when the user clicks the (i) button on a score chip or panel. Generated server-side so the explanation is consistent with the actual computed numbers.
CodiviumRecommendedActions.BuildFromKatex()One recommended action per dashboard panelSelects the most actionable next practice based on the computed data (e.g. "your convergence in Trees is weak and you have invested time there → recommend a Trees session"). This is the analysis that drives the CTA buttons.
Section 12

How to call the aggregation functions

The following is a complete example of going from DB query results to a finished DashboardPayloadV2. In production this logic lives inside your dashboard API endpoint handler.

Step 1 — Build raw data from DB rows

// DB query results (your repositories supply these)
IEnumerable<CodingSessionRow> sessionRows = await _repo.GetCodingSessionsAsync(userId);
IEnumerable<CodingAttemptRow> attemptRows = await _repo.GetCodingAttemptsAsync(userId);
IEnumerable<McqAttemptRow> mcqRows    = await _repo.GetMcqAttemptsAsync(userId);
int uAll = await _repo.GetUniqueCorrectMcqCountAsync(userId);

// Load scoring config (once at startup, not per request)
CodiviumScoring.ScoringConfig cfg = _scoringConfigService.Get();

// Build the raw input object
var rawData = CodiviumRawDataBuilder.Build(
    config:                  cfg,
    codingSessions:          sessionRows,
    codingAttempts:          attemptRows,
    mcqAttempts:             mcqRows,
    uniqueCorrectMcqAllTime: uAll);

Step 2 — Build the payload

var toUtc   = DateTimeOffset.UtcNow;
var fromUtc = toUtc.AddDays(-120);  // last 120 days for the time-on-platform chart

var payload = DashboardPayloadV2Adapter.BuildFromRaw(
    rawData:              rawData,
    fromUtc:              fromUtc,
    toUtc:                toUtc,
    granularity:          "day",
    sessionStartEndpoint: "/api/sessions/start",
    uiPrefs:              uiPrefs,    // DashboardUiPrefs from your prefs store (nullable)
    clock:                null);      // null = use DateTimeOffset.UtcNow

Step 3 — Serialise and return

var json = JsonSerializer.SerializeToUtf8Bytes(payload, new JsonSerializerOptions
{
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
    WriteIndented = false
});
return Results.Bytes(json, "application/json");
Performance note

BuildFromRaw() is CPU-bound and synchronous. For a user with 500 sessions it typically completes in under 50ms. For very large histories consider caching the result for 60–120 seconds (see Section 18).

Section 14

CTA buttons — what they are and how they work

Each dashboard panel has a CTA (Call To Action) button in its bottom-right corner. These buttons are the mechanism by which the dashboard closes the deliberate practice loop: the analytics identify what the learner should do next, and the button takes them directly into a practice session for that recommendation.

What the backend computes

CodiviumRecommendedActions.BuildFromKatex() analyses the payload data and selects one recommended action per panel. Each action has:

FieldExampleMeaning
id"cta_depth_raise_trees_micro"Stable identifier for analytics tracking.
panelId"depth"Which panel this CTA belongs to.
label"Deepen: Trees"Button text shown to the user.
actionType"start_session"Always "start_session" in current implementation.
track"micro"Which track the session should be on.
params.category"Trees"Display name of the recommended category.
params.categoryId"trees"Canonical category key (lower+trim).
params.intent"raise_depth"Practice intent. One of: raise_depth, broaden_coverage, improve_convergence, rebalance, general.
params.count5Recommended number of exercises.
params.timeboxMinutes35Recommended session length in minutes.

What happens when the user clicks a CTA

The dashboard makes a POST request to meta.sessionStartEndpoint (which you set in BuildFromRaw()). The request body is a JSON object containing the action fields above plus "source": "dashboard". On a successful response (HTTP 200) the dashboard reads the redirectUrl field from the response body and navigates to it.

Section 15

Handling CTA calls

Your application needs a POST /api/sessions/start endpoint (or whatever URL you configure as sessionStartEndpoint) that accepts the following request body:

{
  "actionId":   "cta_depth_raise_trees_micro",
  "panelId":    "depth",
  "track":      "micro",
  "category":   "Trees",
  "difficulty": "intermediate",
  "source":     "dashboard",
  "params": {
    "category":      "Trees",
    "categoryId":    "trees",
    "intent":        "raise_depth",
    "count":         5,
    "timeboxMinutes": 35
  }
}

And it must return:

{
  "sessionId":   "abc123...",  // your new session ID
  "redirectUrl": "/practice/start?sessionId=abc123"
}

The reference implementation in api/Program.cs shows the minimal stub. In production this endpoint needs to do real work — see Section 16.

Reference C# endpoint

app.MapPost("/api/sessions/start", async (
    HttpContext http,
    IUserContext user,
    StartSessionRequest req,
    ISessionService sessions) =>
{
    string userId = user.GetRequiredUserId(http);

    // Log for analytics
    _logger.LogInformation(
        "CTA: user={U} action={A} track={T} category={C} intent={I}",
        userId, req.ActionId, req.Track, req.Category,
        req.Params?.GetValueOrDefault("intent"));

    // Create session in your application
    var session = await sessions.CreatePracticeSessionAsync(new CreateSessionRequest
    {
        UserId      = userId,
        Track       = req.Track,
        CategoryId  = (string)req.Params["categoryId"],
        Intent      = (string)req.Params["intent"],
        Count       = (int)((JsonElement)req.Params["count"]).GetInt32(),
        TimeboxMins = (int)((JsonElement)req.Params["timeboxMinutes"]).GetInt32()
    });

    return Results.Json(new { sessionId = session.Id, redirectUrl = session.Url });
});
Section 16

Making CTAs fully functional

The dashboard computes the recommendation (category, track, intent, count, timebox) but cannot resolve this into specific exercises because it has no exercise catalogue. Your application must supply that resolution. Here is everything that needs to exist:

What your application needs to provide

  1. Exercise catalogue. A queryable list of exercises with stable IDs, category, track, and difficulty. This does not flow through the dashboard at all — it lives entirely in your application layer.
  2. Session creation service. Given a user ID, track, category, intent, count, and timebox, selects count exercises from the catalogue, applies a "not recently solved" filter (to avoid immediately re-serving exercises completed in the last few days), and creates a session record in your DB.
  3. Practice UI. The page or view that the redirectUrl points to — the actual coding or MCQ interface where the learner completes the session. When a session is completed, it writes the results back to your DB (new coding_sessions and coding_attempts rows).
  4. Evidence loop closure. The next time the dashboard is loaded for this user, the new session data will be in the DB. BuildFromRaw() will pick it up, scores will update, and the dashboard will reflect the new state.

Exercise selection logic (recommended)

The intent field tells you why this category was recommended. Use it to decide which exercises to serve:

IntentRecommended exercise selection strategy
raise_depthPick exercises in the category at the learner's current competence level or one step above. Prefer exercises not solved in the last 14 days.
broaden_coveragePick exercises in a category the learner has little or no evidence in. Start with Basic difficulty.
improve_convergencePick exercises in the category where the learner's heatmap shows poor A1/A2 quality. Include a mix of difficulty levels.
rebalancePick exercises spread across several under-practiced categories, not just one.
generalPick exercises across the learner's weakest categories. Used for the time-on-platform CTA.
On difficulty selection

The CTA params object may include a difficultyMix field for some panel types (e.g. the heatmap CTA). If present, use it. If not present, default to the learner's current primary difficulty level as inferred from their most recent accepted solves.

Redo spacing

The KaTeX scoring model counts a redo as valid only when spaced at least 7 days from the previous solve. When selecting exercises for a session, filtering out exercises solved in the last 7 days serves two purposes: it avoids serving exercises that feel repetitive to the learner, and it ensures that when the learner does re-solve them later, those re-solves generate valid redo points.

Section 17

UI preferences endpoint

The dashboard allows the learner to change their layout (which panels are visible, which layout preset is active). These preferences are sent to POST /api/dashboard/ui-prefs whenever the user makes a change, and are loaded back at the start of the next session via the meta.ui field of the payload.

You need to:

  1. Implement POST /api/dashboard/ui-prefs to save the preferences object for the user. The reference implementation uses a JSON file store (JsonFileDashboardUiPrefsStore). In production, store in your DB against the user ID.
  2. When building the payload, load the stored preferences and pass them as uiPrefs to BuildFromRaw(). The adapter embeds them in meta.ui.

If you do not implement this endpoint, preferences will still work locally within a session (the dashboard stores them in localStorage) but will not persist across devices or be cleared on login.

Section 18

Caching

The payload is expensive to compute for users with large histories (many sessions × many categories). Cache the serialised JSON bytes, not the DTO object, to avoid re-serialisation on cache hits.

var cacheKey = $"dash:v2:{userId}:{fromUtc:yyyyMMdd}:{toUtc:yyyyMMdd}";
if (cache.TryGetValue(cacheKey, out byte[] cached))
    return Results.Bytes(cached, "application/json");

// ... build payload ...
cache.Set(cacheKey, json, new MemoryCacheEntryOptions
{
    SlidingExpiration = TimeSpan.FromSeconds(60)
});

Invalidate the cache when the user completes a new session (so the next dashboard load shows fresh data) and when the user changes their UI preferences (so the layout is up to date).

Cache key must include the user ID

Never cache a payload without the user ID in the key. The payload contains the user's personal scores, session data, and recommended actions — it must not be served to a different user.

Section 19

Security and authentication

The dashboard has no authentication logic of its own. All three endpoints (/api/dashboard/payload, /api/sessions/start, /api/dashboard/ui-prefs) must be protected by your application's existing authentication middleware.

  • Authenticate before querying. The payload endpoint queries DB data for a specific user. Your middleware must resolve the authenticated user ID from the session cookie or JWT before the handler runs. The reference IUserContext.GetRequiredUserId() is a placeholder — replace it with your real auth resolution.
  • Never trust userId from the request body. The dashboard does not send a user ID — the server derives it from the authenticated session. This prevents IDOR attacks where a user requests another user's payload.
  • CORS. If your dashboard page and your API are on different origins (e.g. embedding via iframe), configure CORS on the API to allow requests from the dashboard's origin.
  • CSP. The dashboard uses inline styles (for dynamic chart colours) and loads vendor scripts (Chart.js, KaTeX). Your Content-Security-Policy must allow these. See csp/ in the package for reference header values.
  • No PII in the payload. The payload contains scores and session statistics, but not names, emails, or other PII. Do not add PII fields to the payload shape.
