Codivium Platform — Developer Reference

Developer Reference

Version 1.1.0
Last updated 2026-03-19
Status Current — single authoritative reference

This is the single consolidated developer reference for the Codivium platform frontend package. It replaces the previous collection of individual Markdown documents that were distributed separately across the package root. Every piece of documentation that was previously spread across 19 separate files has been gathered here, reorganised into a coherent reading order, and presented with navigable sections.

What this document is for

This document serves three distinct audiences who arrive at the package at different stages:

A new developer receiving the package for the first time needs to understand quickly what is in it, what is ready to use immediately, what still needs backend wiring, and how to get the checks running. For this person, start with Quick Start and Package Status & Handover.

A backend developer integrating the package into a live application needs to know exactly which endpoints to implement, what shape the API expects and returns, and how the frontend connects to the backend at each step. For this person, Integration Guide is the primary section, supported by Deployment Guide and the contracts in contracts/.

A developer maintaining or extending the package over time needs to know which files to edit and which to leave alone, what the release criteria are, what the known limitations are, and what is deliberately deferred. For this person, Edit Guidance, Governance, and Known Limitations are the ongoing reference.

Section 01

Quick Start

The minimum steps to verify the package is healthy and to view each product.

Prerequisites

node --version   # must be >= 18.0.0

No other runtime dependencies. All checks run in vanilla Node.js.

Run all checks

cd codivium_platform
bash scripts/run_checks.sh

Expected output: All checks passed. Individual checks are also available:

npm run test        # 49 JS unit tests
npm run smoke       # integration smoke test
npm run validate    # payload contract validation
npm run security    # CSP / inline checks
npm run links       # broken link detection

Common questions

Can I deploy editor.html today?

No. The backend endpoints are not implemented. See Package Status & Handover §Editor.

The link checker is failing

Run node scripts/link_check.js for details. Most failures are missing .html stub files.

The security gate is failing

Run bash scripts/security_gates.sh for details. Usually an inline <script> or style= attribute. All JS goes in editor-page.js; all CSS in editor-page.css.

How do I rebuild the dashboard bundle?

bash scripts/build_dashboard_bundle.sh
bash scripts/run_checks.sh

Where does state live?

See assets/components/editor/editor-page.state-model.js for all five state layers.

What this package gives you

Step 1 — Exercise database

Create an Exercises table with the fields in contracts/exercise-workspace-api.contract.md. Seed with your exercise content.

Step 2 — Menu endpoint

Implement GET /api/menu/payload?track={trackId}. Return shape:

{
  "track": "micro",
  "trackLabel": "Micro Challenges",
  "categories": ["Arrays", "Strings", "Stacks"],
  "difficultyLevels": ["beginner", "intermediate", "advanced"],
  "exercises": [
    {
      "id": "ex_min_window_001",
      "name": "Minimum Window Substring",
      "shortDescription": "Find the smallest substring...",
      "category": "Strings",
      "difficulty": "advanced",
      "completionStatus": "not_started",
      "completedAt": null
    }
  ]
}

completionStatus is per-user. Query your submissions table.

Step 3 — Exercise detail endpoint

Implement GET /api/exercise?id={exerciseId}. Full contract: contracts/exercise-workspace-api.contract.md.

{
  "id": "ex_min_window_001",
  "name": "Minimum Window Substring",
  "category": "Strings",
  "difficulty": "advanced",
  "testsTotal": 24,
  "problemStatement": "Given strings s and t...",
  "codeScaffold": "def minWindow(s, t):\n    pass\n",
  "hints": "**Hint 1:** Consider sliding window...",
  "priorAttempts": 0,
  "bestPriorScore": null,
  "lastSolvedAt": null
}

Step 4 — Submit endpoint

Implement POST /api/exercise/submit.

// Request
{
  "exerciseId": "ex_min_window_001",
  "code": "def minWindow(s, t):\n    ...\n",
  "sessionId": "sess_lf3kj2_a7c9m1",
  "elapsedSecs": 847,
  "attempt": 3
}

// Success response
{
  "accepted": true,
  "testsPassed": 24,
  "testsTotal": 24,
  "failedTests": [],
  "feedback": {
    "exerciseName": "Minimum Window Substring",
    "category": "Strings",
    "difficulty": "advanced",
    "isFirstSolve": false
  }
}

Step 5 — Connect submission to dashboard

When a submission is accepted, create a CodiviumScoring.CodingSession record so the dashboard reflects completed exercises:

var session = new CodiviumScoring.CodingSession
{
    ExerciseId     = req.ExerciseId,
    TrackId        = exercise.TrackId,
    Category       = exercise.Category,
    StartedAtUtc   = DateTimeOffset.UtcNow.AddSeconds(-req.ElapsedSecs),
    CompletedAtUtc = DateTimeOffset.UtcNow,
    UnitTestsTotal = result.TotalCount,
    Attempts       = /* build from submission history */
};

CSRF

If your deployment uses CSRF tokens, inject before editor-page.js loads:

<script>
  window.__CODIVIUM_CSRF_TOKEN = "{{ csrf_token }}";
</script>

The submit module automatically includes it as X-Csrf-Token.

JWT authentication

All API calls use Authorization: Bearer <token>. The token is read from localStorage['cv_auth_token'] via getAuthHeaders() in global.js. Set the token after successful login:

localStorage.setItem('cv_auth_token', yourJwtToken);

On the server, register JWT middleware in Program.cs: services.AddAuthentication().AddJwtBearer(...).

Analytics events

The submit module dispatches these events on document:

EventWhen
exercise_loadedExercise data received from API
code_changedUser edits the code editor
submit_clickedSubmit button pressed
submit_successAll tests passed
submit_failSome tests failed
submit_errorNetwork or server error
focus_mode_changedUser changes focus mode

Section 07

Security Hardening

1. Recommended deployment model

Self-host all assets on the same origin. Run scripts/fetch_vendor_deps.sh to vendor Chart.js and KaTeX. When vendor files are present, the CSP can be tightened to script-src 'self'.

2. Content Security Policy

Option A — Self-hosted assets (recommended)

Content-Security-Policy: default-src 'self';
  base-uri 'self';
  object-src 'none';
  frame-ancestors 'self';
  form-action 'self';
  script-src 'self';
  style-src 'self';
  img-src 'self' data:;
  font-src 'self';
  connect-src 'self';

Option B — CDN scripts for KaTeX / Chart.js

Content-Security-Policy: default-src 'self';
  script-src 'self' https://cdnjs.cloudflare.com https://cdn.jsdelivr.net;
  style-src 'self' https://cdn.jsdelivr.net;
  font-src 'self' https://cdn.jsdelivr.net;
  connect-src 'self';
!
CDN fallback is only activated when data-cv-allow-cdn="true" is set on the <html> element. Production pages should omit this attribute.

3. Auth — JWT Bearer tokens

Auth is JWT Bearer token based. The frontend reads the token from localStorage['cv_auth_token'] via getAuthHeaders() in global.js. All API calls include Authorization: Bearer <token>. Register JWT middleware in ASP.NET Core — the user ID is read from ClaimTypes.NameIdentifier or the sub claim.

4. CSRF and session endpoints

For state-changing endpoints, use CSRF tokens via window.__CODIVIUM_CSRF_TOKEN. The submit module includes it as X-Csrf-Token automatically. For JWT-authenticated endpoints, prefer Authorization: Bearer and keep credentials: 'omit'.

5. Recommended HTTP response headers

X-Content-Type-Options: nosniff
Referrer-Policy: no-referrer
Permissions-Policy: geolocation=(), microphone=(), camera=()
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

6. Redirect handling

The frontend normalizes and restricts redirects from recommended actions. Server-side: treat redirectUrl as untrusted input. Whitelist allowed paths. Reject javascript:, data:, and cross-origin URLs.

7. Input validation

Validate IDs, category keys, and numeric ranges in raw scoring input. Keep text as plain text — avoid rendering untrusted HTML.

8. SRI hashes

Editor CDN assets (CodeMirror, marked) are marked data-sri-pending="true". Run scripts/verify_sri.sh at deployment time to generate and embed the hashes.

Section 08

Operational Readiness Checklist

Observability

  • Client-side error capture wired to your telemetry (Sentry / Datadog / etc.)
  • Key API endpoints have request IDs and structured logs
  • Dashboards: error rate, latency, p95/p99, timeouts

Security

  • Strict CSP deployed for dashboard pages
  • Clickjacking protection (frame-ancestors)
  • CSRF protections for state-changing endpoints
  • Rate limits for session start / payload endpoints
  • Secrets stored in a secret manager; rotation plan exists
  • JWT middleware registered; GetUserId(ctx) reads from NameIdentifier / sub claim
  • SRI hashes generated via scripts/verify_sri.sh and embedded in editor.html

Reliability

  • Blue/green or canary releases for frontend changes
  • Rollback procedure documented and tested
  • Static asset caching strategy verified

Data protection

  • Backups configured and restore test performed
  • Audit logs for login / subscription / admin actions
Section 09

Threat Model

Scope: Codivium Insights Dashboard page.

Assets

  • Authenticated user session (JWT tokens)
  • User metrics payloads and insights
  • Subscription entitlements (authorization)
  • Admin/config endpoints (if applicable)

Trust boundaries

Browser ↔ API ↔ datastore; external vendor libraries if loaded from CDN.

Entry points

  • HTML entry: codivium_insights_embedded.html
  • Scripts: assets/components/**
  • API: /api/sessions/start and payload endpoints
  • Redirect / navigation actions (recommended actions)

Key threats and mitigations

ThreatMitigation
XSSAvoid innerHTML for untrusted strings; enforce strict CSP; escape server outputs
CSRFJWT Bearer tokens; X-Csrf-Token header for state-changing endpoints
IDORServer-side authorization checks on all resource IDs
Open redirectRestrict redirects to same-origin / allowlist — implemented in dashboard
ClickjackingSet CSP frame-ancestors appropriately
Supply chainPrefer self-hosted vendor libs; avoid uncontrolled CDN fallback in production; SRI hashes
Abuse / rate limitingProtect expensive endpoints with rate limits and caching

Monitoring signals

  • Spike in /api/sessions/start errors
  • Spike in client-side JS exceptions
  • Unusually frequent requests from a single IP / account

Performance Guide

Reduced motion

When the OS requests reduced motion (prefers-reduced-motion: reduce), all animations and transitions are minimized across the dashboard and editor. The MCQ answer-reveal delay collapses from 2400ms to 600ms.

Low-effects mode

Reduced visual effects mode for low-end devices:

// Query param
?cvEffects=low

// Programmatically
CodiviumInsights.setEffectsMode('low')

Low-effects mode disables heavy overlays, blur filters, and reduces chart animation duration.

Font loading

Cinzel brand font loads from Google Fonts. The crossorigin="anonymous" attribute is present on all Cinzel <link> tags to enable the preconnect hint. For a strict CSP that blocks external font origins, vendor the font files locally.

Offline & Vendor Mode

Status: Supported

Offline mode is officially supported. Required vendor files:

  • assets/vendor/chartjs/chart.umd.min.js
  • assets/vendor/katex/katex.min.js
  • assets/vendor/katex/katex.min.css
  • assets/vendor/katex/contrib/auto-render.min.js
  • assets/vendor/katex/fonts/*.woff2 (20 font files)

To enable offline mode

bash scripts/fetch_vendor_deps.sh

CDN fallback behaviour

polyfill-loader.js uses a local-first strategy:

  1. Try assets/vendor/... (local)
  2. If missing → try CDN (only if data-cv-allow-cdn="true" is set on the <html> element)

Demo pages have data-cv-allow-cdn="true" for convenience. Production pages should omit this attribute.

Smoke test warnings

Before running fetch_vendor_deps.sh you will see:

WARN: Chart.js vendor file missing: run ./scripts/fetch_vendor_deps.sh for offline mode
WARN: KaTeX vendor files missing: run ./scripts/fetch_vendor_deps.sh for offline mode

These are pre-deployment warnings, not errors. They do not affect the check suite result.
