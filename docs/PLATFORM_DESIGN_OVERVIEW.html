Section 1

What Codivium is

Codivium is a deliberate practice platform for software engineers. It presents algorithmic coding exercises and knowledge-check quizzes, tracks the quality and breadth of a learner's practice over time, and produces a mastery score that reflects not just how much they have practised, but how well and how broadly.

The platform is designed around the principle that mastery requires two things simultaneously: breadth (having practised across a wide range of skill categories) and depth (having practised any given category enough times, at high enough quality, that the evidence of competence is substantial). A learner who has solved 200 Array problems but never touched Trees has high depth in one area and low breadth overall. A learner who has touched every category once has high coverage but no depth anywhere. The Codivium Score rewards both.

The platform is built for post-login use. Every page assumes an authenticated user. The package described in this document contains all the post-login pages and the shared component library they use.

Section 2

The three tracks

All practice activity is categorised into one of three tracks. Each track is a distinct mode of activity with its own scoring contribution.

Micro Challenges

Short, focused algorithmic coding exercises. The learner is given a problem, writes code to solve it, and submits. The code is tested against an automated unit-test suite. A session is accepted when all unit tests pass. Micro challenges are designed to be completable in 15–45 minutes and test one narrow skill at a time. The canonical track_id is "micro". Currently 14 skill categories are configured.

Interview Preparation

Longer, more complex coding exercises that simulate real technical interview problems. They use the same accepted/failed mechanism as micro challenges but at greater difficulty and scope. The canonical track_id is "interview". Same 14 categories as micro.

MCQ (Multiple Choice Quizzes)

Knowledge-check quizzes. A quiz is a set of multiple-choice questions on a single category at a single difficulty level. Unlike coding tracks, MCQ has no pass/fail — every completed quiz attempt contributes evidence proportional to the score percentage achieved. The canonical track_id is "mcq". Same 14 categories.

The 14 skill categories

Arrays, Strings, Stacks, Searching, Sorting, Linked Lists, Recursion, Dynamic Programming, Greedy Algorithms, Graphs, Heaps, Tries, Trees, Mathematical. These categories are the denominators in all breadth calculations and the axes of all charts. Adding a new category requires updating scoring-config.v1.json before it is recognised by the scoring engine.

Section 4

Shared components

Every page in the package loads the same set of shared CSS and JS files. No page defines its own topbar, sidebar, logo, or cube — they all consume these shared components, which means any change to a shared file affects all pages simultaneously.

CSS load order (all pages)

  1. assets/components/core/base.css — CSS custom properties (design tokens), body reset, watermark utility classes, effects-mode controls.
  2. assets/components/logo/logo.css — The 3D cube logo: face geometry, .cube-face transforms, gold border, symbol styles, animation keyframes.
  3. assets/components/topbar/topbar.css — Fixed top navigation bar: layout, background gradient, nav links, active-page indicator pill.
  4. assets/components/sidebar/sidebar.css — Left sidebar: container, handle, side-nav, side-link base styles, collapsed state, profile card.
  5. assets/components/sidebar/menu-overrides.css — Platform-level overrides: canonical design tokens, sidebar button shape (square, 64px height, flat gradient), side-nav vertical positioning (padding-top: 40px, gap: 43.6px), watermark position classes wm1–wm7.
  6. Page-specific CSS — loaded last, after all shared files.

JS load order (all pages)

  1. assets/components/logo/brand-cube.js — WAAPI cube hover animation: on mouseenter starts a progressive tumble spin; on mouseleave smoothly returns to rest pose. Respects prefers-reduced-motion.
  2. assets/components/sidebar/sidebar.js — Sidebar collapse/expand, drag-to-collapse gesture, active-page indicator management (reads data-page on body load, updates on hash-link clicks), disabled-link guard.
  3. assets/components/core/global.js — Hash-link click guard, data-tip to title copy, effects-mode (low/full) read/write via localStorage and URL param.

The logo and cube

The animated cube in the top-left corner of every page is the Codivium brand mark. It is a CSS 3D cube with six faces, each displaying a Greek letter (κ, δ, φσα, η, ν, δα) that reference the KaTeX scoring model's mathematical symbols. The cube rests at a fixed angle (rotateX(-18deg) rotateY(32deg)) and spins continuously while the user hovers over the brand area. On mouseleave it smoothly returns to the rest pose via a WAAPI animation. The faces are fully opaque with a gold border and radial highlight gradients. Flickering was eliminated by setting transform-style: flat on faces (preventing children from entering the 3D scene as independent depth-sorted planes).

Sidebar button positioning

Both existing pages use identical button geometry, sourced from menu-overrides.css: height 64px, border-radius 0 (square), flat dark gradient background, inset top highlight and bottom shadow. The side-nav padding-top is 40px (reduced from the sidebar.css default of 168px to raise buttons closer to the topbar), with a gap of 43.6px between buttons. This gap was calculated to keep the MCQ button at exactly the same vertical position as the default layout while raising the first button by two button-heights.

Section 5

The scoring model (KaTeX)

The KaTeX scoring model is the mathematical engine that converts raw practice data into the metrics shown on the dashboard. All computation happens server-side in CodiviumKatexScoring.v1.cs. The scoring is completely stateless: it takes a snapshot of all historical session data and produces scores fresh on every call. No scores are stored in the database.

Evidence

Evidence is the core currency of the scoring model. Every accepted coding session and every MCQ quiz attempt produces a quantum of evidence in a specific category. Evidence accumulates over time but decays with age.

For a coding session, evidence is: E = f_d × q_tries × q_conv × q_time × decay

  • f_d — difficulty factor: basic = 1.0, intermediate = 1.3, advanced = 1.6. Harder exercises produce more evidence.
  • q_tries — convergence quality based on attempt count: clamp(1/√attempts, 0.45, 1.0). First-try solves produce maximum quality. Many attempts reduce quality but it never drops below 45%.
  • q_conv — bucket weight: A1 (1 attempt) = 1.00, A2 = 0.92, A3 = 0.85, A4 = 0.78, A5 = 0.70, A6+ = 0.60. Separate from q_tries; rewards quick convergence directly.
  • q_time — time quality: clamp(exp(-t/T_d), 0.6, 1.0) where T_d = 90/120/160 minutes for basic/intermediate/advanced. Fast solves earn slightly more evidence; the floor of 0.6 means slow solves are never heavily penalised.
  • decay — recency weight: 0.5^(days/15). Evidence from 15 days ago counts half as much as today's. Evidence from 30 days ago counts a quarter.

For MCQ, evidence is: E = (ScorePercent/100)^1.5 × f_d × decay. The 1.5 exponent means higher scores contribute disproportionately more.

Breadth

Breadth measures how widely spread practice is across all categories in a track. It combines three sub-components:

  • Coverage — the fraction of categories that have reached the activation threshold τ_E = 1.5 evidence. A category below τ_E is not yet considered genuinely covered.
  • Balance — the entropy of evidence distribution across categories, normalised to [0,1]. Concentrating all practice in one category produces low balance even at high coverage.
  • Confidence — a stabiliser: 1 - exp(-ΣE/k) where k is a track-specific constant (micro k=10, interview k=8, mcq k=14). New users with little evidence score conservatively even if their few sessions happened to cover many categories.

Breadth = 100 × Confidence × (0.65 × Coverage + 0.35 × Balance)

Overall breadth is a weighted average: micro 40%, interview 40%, MCQ 20%.

Depth

Depth measures how much evidence has accumulated within each category, using a saturating curve so that early practice produces fast gains and further practice produces diminishing returns. Per category: D = 100 × (1 - exp(-E/λ)) where λ = 12. Overall depth is the evidence-weighted average across all categories — categories where the learner has done more work contribute more to the overall figure.

Codivium Score

The headline mastery number. It is the harmonic mean of overall breadth and overall depth: Score = 2BD/(B+D). The harmonic mean means weakness in either component pulls the score down significantly. A learner with breadth 80 and depth 20 scores 32, not 50. Both components must be developed together to score well.

Points

Points are a career progression metric, separate from mastery. They reward volume and consistency.

  • First-solve points: P0 = 4.0 per unique exercise, multiplied by difficulty factor and convergence quality. Earned once per exercise (the first time it is accepted).
  • Redo points: P_redo = 2.5 × ln(1 + R_valid) per exercise, where R_valid counts spaced re-solves. A re-solve only counts if at least 7 days have passed since the previous solve of the same exercise. The logarithm prevents gaming by grinding one exercise repeatedly.
  • MCQ points: P_mcq × √U_all where U_all is the lifetime count of distinct MCQ questions ever answered correctly. The square root keeps growth controlled over long careers.

The 30-day points variant counts only first-solves and valid redos whose completion timestamp falls within the last 30 days. This is a hard window, not a recency-decayed sum.

Efficiency

Efficiency measures quality-weighted points earned per hour of practice, with recent sessions weighted more heavily. Formula: η = Σ(w_i × q_i) / (ε + Σ(w_i × t_i)), displayed as η × 60 pts/hr. Here w_i is the recency weight, q_i = f_d × q_tries × q_conv (note: no q_time, to avoid double-counting time in both numerator quality and denominator time), t_i is time-to-AC in minutes, and ε = 1 minute is a small epsilon to prevent division by zero. Efficiency rewards fast, clean, quality solves and penalises slow or many-attempt sessions via the time denominator.

Section 6

Performance Insights dashboard

The dashboard is the primary mastery reporting page. It shows the learner their current state across all metrics and all tracks, and provides specific recommendations for what to practise next. It is a pure rendering layer — it receives a single JSON payload from the server and displays it. All computation happens on the server before the payload is delivered.

Info pane insights

Every score chip and every panel has an associated insight object pre-computed by CodiviumInsights.v1.cs. When the learner clicks the (i) button on any metric, a sliding info pane opens showing a personalised analysis of that specific metric. The analysis is rule-based (not AI-generated) and uses the same data the scoring engine used to produce the metric value.

Each insight has: a title, a summary sentence, and multiple sections. Sections contain paragraphs, bullet lists, and call-out notes. The insight for a given metric names the learner's actual values, identifies which specific lever (time vs attempts vs category distribution) is most affecting the metric, and gives 2–3 concrete actions. The insight builder for each metric is a private static method in CodiviumInsights.v1.cs — for example BuildDepthInsight(), BuildEfficiencyInsight(), BuildFirstTryInsight().

CTA buttons

Each panel has a CTA (Call To Action) button at the bottom right. The backend analyses the payload data and selects the most actionable recommendation for that panel. When the learner clicks the button, the dashboard POSTs to meta.sessionStartEndpoint (configured as /api/sessions/start) with a structured params object containing the recommended track, category, intent, count, and timebox. The server creates a session and returns a redirectUrl. The browser navigates to that URL.

The recommendation logic lives in CodiviumRecommendedActions.v1.cs. It selects the category and intent based on the computed data — for example, the heatmap CTA picks the category with the worst A1/A2 convergence; the depth CTA picks the category with the most room to grow; the time allocation CTA picks the most under-practised category. The exercise catalogue itself is not in the dashboard — the session start endpoint on the application side resolves the recommendation into specific exercise IDs.

Payload structure

The dashboard payload is assembled by DashboardPayloadV2Adapter.BuildFromRaw() and delivered via GET /api/dashboard/payload. It is a single JSON object with six top-level keys: meta (endpoint config, anchor date, UI prefs), overall (all scalar metrics, daily time series, insight dictionary, recommended actions), combinedCoding (depth-by-category and heatmap for micro + interview merged), micro (same, micro only), interview (same, interview only), mcq (MCQ metrics and chart data). The frontend receives this and renders everything from it — there are no subsequent API calls during a dashboard session.

Section 7

Exercise menu page

The menu page shows the learner all available exercises for a single track, with filtering, sorting, and search. It is the bridge between the dashboard's recommendations and the actual coding work. One HTML file (menu-page.html) serves both the Micro Challenges and Interview Preparation tracks via a URL parameter.

Data and filtering

The page reads ?track=micro or ?track=interview from the URL and fetches a menu payload from GET /api/menu/payload?track=.... The payload contains: the full list of exercises for that track (id, name, category, difficulty, short description, completion status), the list of all available categories, and the list of all difficulty levels. The completion status for each exercise is joined server-side from the user's session history.

All filtering happens client-side in memory. The full list is delivered on load and never re-fetched during a browse session. The filter drawer supports filtering by category (multi-select), difficulty level (multi-select), and completion status (All / Completed / Not Started). The sort controls support A–Z, by category, by difficulty, and by completeness. A text search field filters on name and short description simultaneously. All filters, sort, and search are combined via AND — a result must satisfy all active constraints.

At the expected scale of 200–1,000 exercises, no virtual scrolling or pagination is required. The full list renders into the CSS grid on load. Text search is added from the start because browsing a 1,000-item grid without being able to type is impractical.

Cards and navigation

Each exercise is rendered as a card in a 6-column grid. Each card shows the exercise name, a short description, category pill, difficulty pill, and a completion status indicator (green tick for completed, yellow cross for not started). Clicking a card navigates to the editor page:

editor.html?id=micro_042&from=menu-page.html%3Ftrack%3Dmicro

The from parameter encodes the return URL. The editor page uses this to construct the "Return to Menu" destination in the feedback modal after a successful completion. This means the browser back button and the explicit "Return to Menu" button both send the user to the same place — the correct track's menu page.

Exercise IDs are globally unique across all tracks. If the micro and interview exercise databases use independent ID sequences, a track prefix is prepended: micro_042 for micro challenge #42, interview_042 for interview exercise #42. This ensures the scoring engine's distinct-solve counting and redo-spacing logic operates correctly.

Search field

A text input field, persistent above the exercise grid (not inside the collapsible filter drawer), filters the exercise list in real time as the user types. It matches against the exercise name and short description using case-insensitive substring matching. The search result is combined with any active filter drawer selections via AND. No server call is made; the search operates entirely over the in-memory exercises array.

Section 8

Editor page (pending)

The editor page is the coding environment for a single exercise. It has not been built yet. The design decisions that have been made are:

  • URL contract: editor.html?id={exerciseId}&from={encodedReturnUrl}. The exercise ID identifies what to load. The from parameter is the full return URL (e.g. menu-page.html?track=micro) encoded as a query parameter.
  • Data fetch: On load, the page calls GET /api/exercise?id={exerciseId} which returns the full problem statement, constraints, examples, test suite definition, code scaffold, and the user's previous attempts for this exercise if any.
  • On successful submission: The submission endpoint returns a feedback payload containing all the data needed for the feedback modal (current run stats, historical stats for this exercise, delta vs previous completion). The editor calls CodiviumFeedback.show(result) immediately. The feedback modal appears as a floating overlay on top of the editor — the user can still see their solution behind the modal.
  • On failed submission: No dialog. The editor stays open. The user's solution and the test results remain visible. The user can keep trying, use the sidebar to navigate somewhere else, or click the browser back button to return to the menu.
  • The sidebar is always present: The editor page has the full shared sidebar. Navigation via the sidebar is always available, including mid-exercise.
Design rationale

The asymmetry between success and failure handling is deliberate. Success deserves a moment of ceremony and a clear decision point. Failure deserves frictionless continuation — forcing a dialog on every failed submission would be actively disruptive to the flow state required for problem-solving.

Section 9

Feedback modal

The immediate feedback modal appears when a coding exercise is successfully completed (all unit tests pass). It is a floating overlay — the editor and the user's solution remain visible behind it. It is 1040px wide (matching the original concept design) with a semi-transparent blurred backdrop. There is no close button. The only exit paths are the three navigation buttons in the footer.

Content rows

The modal body contains four rows:

Row 1 — Primary metrics + Insight

Two primary metric cards side by side on the left (Time to Success with mm:ss format, and Attempts as a count), each with a coloured delta badge showing the percentage change vs the previous completion of this exercise (green for improvement, red for regression). On the right, the Insight panel — a gold-bordered block containing the rule-based analysis text and badge chips.

Row 2 — Secondary micro-metrics

Three smaller cards in a row: Time to First Attempt (how long before the learner made their first submission — a measure of planning vs diving in), Best Pre-Success pass rate (the highest percentage of tests passed by any non-final submission — measures how close the initial solution was), and Average Iteration time (mean time between submissions — measures the efficiency of the debugging loop).

Row 3 — Trend charts

Two charts showing historical data for this specific exercise across the learner's previous completions (up to the last 8). The left chart is a line chart with area fill showing Time to Success trend. The right chart is a bar chart with overlay line showing Attempts trend. Both show Best / Median / Latest in a legend below. This row only renders when the history array contains at least 2 prior completions.

Footer

Left side: a stamp reading "Codivium — Immediate feedback". Right side: three buttons.

Rule-based insight engine

The insight text is generated by a rule set in feedback.js, mirroring the pattern used in CodiviumInsights.v1.cs on the backend. It evaluates the current run and produces a personalised 2–3 sentence paragraph. The rules are:

  • Convergence quality: first-try → precise reasoning; 2 attempts → efficient iteration; 3–4 attempts → pattern identified cleanly; 5+ → suggest pre-submission test case writing.
  • Time quality: compared against the KaTeX T_d benchmarks (90/120/160 minutes for basic/intermediate/advanced). Under 30% of T_d → fast, strong mental model. Under 60% → normal range. Under 100% → approaching reference window. Over 100% → recommend timeboxing and post-mortem.
  • Historical trend: if prior history exists and time improved by more than 10%, calls it out explicitly. If this is a first solve, flags it.

The rule-based approach was chosen over AI-generated text specifically to avoid latency. An AI call on every submission acceptance would delay the feedback dialog appearing by 1–3 seconds. The rule engine is synchronous and produces the insight text before the DOM is even built.

The "next exercise" hint (a chip in the insight panel) is also rule-based: if the learner struggled (5+ attempts or over-time), recommend consolidating at the same difficulty. If they excelled (first-try and fast), suggest stepping up difficulty. Otherwise, continue through the catalogue.

Section 12

End-to-end user journey

The following describes the intended flow for a learner using the platform in a typical session.

Arriving at the dashboard

After login, the learner lands on the Performance Insights dashboard. The server has already computed their full KaTeX scores and assembled the payload. The page renders with all metrics, charts, and CTA buttons immediately visible. The learner can see their Codivium Score, how it breaks down into breadth and depth, where their weakest categories are on the heatmap, and what the system recommends they do next via the CTA buttons in each panel.

Using a CTA button

The learner clicks the CTA button on the Convergence Heatmap panel. The button POSTs to /api/sessions/start with a recommendation of "5 intermediate Dynamic Programming exercises, 35 minute timebox, intent: improve_convergence". The server creates a session, selects exercises from the catalogue that match the recommendation and that the learner has not solved recently, and returns a redirect URL. The browser navigates to menu-page.html?track=micro&highlight=micro_089, or directly to the editor depending on the server's implementation choice.

Browsing the menu

Alternatively, the learner navigates to the Micro Challenges menu via the sidebar. The menu page loads with the full exercise catalogue for that track. The learner uses the filter drawer to narrow to Advanced difficulty, unticks everything except Dynamic Programming, and types "subsequence" in the search field. The grid updates instantly to show only matching exercises. The learner clicks one.

Working in the editor

The editor page opens with the problem statement, a code editor, and a test runner. The learner reads the problem, plans a solution, writes code, and submits. The first submission passes 8 out of 12 tests. The editor shows which tests failed without any interruption or dialog. The learner refines the code and submits again. All 12 tests pass.

Seeing the feedback

The feedback modal appears as a floating overlay on top of the still-visible editor. It shows: "Submission accepted — Minimum Window Substring". Time to Success: 18:22. Attempts: 2 (green badge showing improvement vs prior). The insight reads: "Two submissions to acceptance. Your first attempt reached 67% of tests — one iteration closed the gap. Solve time is within the expected range for Advanced." Three buttons at the bottom: Dashboard, Return to Menu, Try Again.

Choosing what to do next

The learner clicks "Return to Menu". They are taken back to menu-page.html?track=micro. The exercise they just completed now shows a green tick. They pick another exercise. This cycle — browse, attempt, complete, return — is the core deliberate practice loop. Eventually they return to the dashboard and see their scores have updated to reflect the new evidence.

Score update timing

The dashboard payload is cached for 60 seconds by user and date. The learner will see updated scores the next time they load or refresh the dashboard after completing an exercise. The cache is invalidated when a new session is recorded. There is no real-time score update in the current design.
