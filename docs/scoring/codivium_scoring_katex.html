Codivium Scoring Equations + Detailed Explanations (KaTeX)

This document is the scoring “source of truth” for Codivium. It describes the exact metrics the dashboard shows, how they are computed from verified user activity, and why each step exists (i.e., the design intent). Open this file in a browser (internet required for the KaTeX CDN).

Introduction. Codivium is designed for elite deliberate practice. The scoring system reflects that by separating: (1) a capped mastery score that answers “how strong are you right now?”, from (2) uncapped progression and efficiency signals that answer “how much verified work have you banked?” and “how efficiently are you converting time into verified mastery?”.

The dashboard therefore shows one headline number called Codivium Score (0–100), plus two supporting numbers: Codivium Points (uncapped, with both all-time and last-30-days views) and Efficiency (pts/hr) (uncapped rate). These work together: the headline score stays interpretable and comparable, while the supporting numbers reward sustained, high-quality practice.
Dashboard Score Overview
One headline score + two supporting numbers (elite design).

What the user sees

The dashboard displays three primary scoring readouts:

  • Codivium Score (0–100): a mastery score, computed as a harmonic mean of breadth and depth.
  • Codivium Points (uncapped): a progression score for verified volume, shown as (a) all-time points and (b) points gained in the last 30 days.
  • Efficiency (pts/hr) (uncapped): a rate metric that rewards producing accepted solves quickly (and penalizes slow/inefficient progress) while using a recency decay so that recent performance matters most.

The key idea is: Codivium Score stays stable and meaningful, while points and efficiency provide “career” and “momentum” context. This is more premium than collapsing everything into one number, because it remains interpretable and hard to game.

Codivium Score (headline mastery score, capped 0–100)
$$\text{CodiviumScore} = \text{Mastery} = \frac{2BD}{B+D}$$

The harmonic mean ensures the score reflects elite balance: if breadth is high but depth is low (or vice-versa), the score is constrained. This prevents “one-trick” profiles from inflating the headline number.

Codivium Points (uncapped progression; shown as All-time and Last-30-days)
$$\text{Points}_{\text{all}} = \text{CodingPoints}_{\text{all}}+\text{RedoPoints}_{\text{all}}+\text{McqPoints}_{\text{all}}$$ $$\text{Points}_{30} = \text{CodingPoints}_{30}+\text{RedoPoints}_{30}$$

Points are intentionally uncapped so progress can continue for years, but they are designed with diminishing returns and anti-farming rules (especially for re-doing the same exercise).

Efficiency (pts/hr) (uncapped rate; accepted solves only; recent work matters more)
$$\text{Efficiency}=\frac{\sum_i w_i\,q_i}{\epsilon+\sum_i w_i\,t_i}\qquad\text{and}\qquad \text{pts/hr}=60\cdot \text{Efficiency}$$

Efficiency is a rate: “verified quality per minute”. It rewards producing accepted solutions efficiently, and it cannot be increased by spending time without producing accepted solves. This avoids rewarding inefficiency.

Tracks and Raw Data Requirements
What must exist in the raw data so the equations are well-defined.

Tracks (three streams)

Codivium has three tracks, each with its own category set: Micro Challenges, Interview Preparation, and MCQ. We compute breadth per track. For mastery depth, we compute depth for the two coding tracks (micro and interview) separately and then combine them; we do not pool categories across tracks. The depth-by-category bar chart uses the union of coding categories (micro + interview) merged by a canonical category key.

The raw data is expected to be track-aware and category-aware, and it must include timestamps to support recency decay and “last 30 days” windows.

Minimum raw fields for coding sessions (accepted solves)

For each accepted coding solve event, we require:

  • ExerciseId — required to count distinct solves and re-dos correctly.
  • Track — micro or interview (coding tracks).
  • CategoryId — which skill category this exercise belongs to.
  • Difficulty — mapped to \(f_d\).
  • CompletedAtUtc — timestamp used for recency and rolling windows.
  • Attempts — used for tries factor \(q_{\text{tries}}\) and convergence \(q_{\text{conv}}\).
  • TimeToACMinutes — used in \(q_{\text{time}}\) and in efficiency denominator.
Minimum raw fields for MCQ

MCQ contributes to breadth via quiz results and to points via an all-time unique-correct counter. MCQ results are always considered “accepted” (there is no accepted/failed notion; every quiz result counts).

For MCQ breadth, we require one record per quiz result event:

  • QuizId — identifies the quiz instance.
  • CategoryId — the MCQ category for this quiz (single category per quiz).
  • Difficulty — one of basic/intermediate/advanced (single difficulty per quiz), mapped to \(f_d\).
  • CompletedAtUtc — timestamp used for recency decay and rolling windows.
  • ScorePercent — the quiz mark in \([0,100]\). This percentage may reflect partial completion.

For MCQ points (all-time padding), the backend additionally provides: UniqueCorrectMcqAllTime \(=U_{\text{all}}\) — lifetime count of distinct MCQ question IDs answered correctly.

Step 1 — Evidence
Convert verified activity into “proof of work” per track and category.

What evidence is and why it exists

Evidence is the atomic unit of the mastery score. Each verified action contributes some evidence to a category. Evidence is designed to be: verified (only accepted solves count for coding), difficulty-aware (harder work counts more), quality-aware (attempts, convergence, and time influence the value), and recency-aware (newer work counts more than old work).

The reason we do this instead of counting “minutes spent” is that elite practice values outcomes. Time can only help the score if it produces verified competence.

Days since event (recency input)
$$\Delta(e)=\frac{\text{now}-t(e)}{86400},\qquad \Delta(e)\leftarrow \max(0,\Delta(e))$$

We turn timestamps into “days ago”. If a timestamp is accidentally in the future, we clamp it to 0 to avoid giving extra credit.

Recency decay with half-life
$$\text{decay}(e)=0.5^{\Delta(e)/H}$$

The half-life \(H\) means: after \(H\) days, identical work counts half as much. This keeps mastery “current”. Default \(H=15\) days, meaning work 30 days old is weighted at \(0.25\).

Premium coding outcome: accepted solves only
$$o(e)=\mathbf{1}_{\text{accepted}}(e)\cdot q_{\text{tries}}(e)\cdot q_{\text{conv}}(e)\cdot q_{\text{time}}(e)$$

This is the core “premium” rule: unsolved attempts do not contribute. We only count accepted work. The quality factors then distinguish fast, clean solves from slow, many-attempt solves.

Quality factors (defaults)
$$q_{\text{tries}}(e)=\text{clamp}\!\left(\frac{1}{\sqrt{\text{attempts}(e)}},\,0.45,\,1.0\right)$$ $$q_{\text{conv}}(e)=g(\text{A}(e))\quad\text{where}\quad g=[1.00,0.92,0.85,0.78,0.70,0.60]$$ $$q_{\text{time}}(e)=\text{clamp}\!\left(e^{-t(e)/T_d},\,0.6,\,1.0\right)$$

Explanation:

  • tries penalizes repeated submissions (diminishing returns).
  • convergence uses attempt buckets: \(A(e)=\min(\text{attempts}(e),6)\), where A1..A5 correspond to 1..5 attempts and A6 corresponds to >5 attempts.
  • time uses \(t(e)=\text{TimeToACMinutes}(e)\) with a mild bounded penalty. \(T_d\) depends on difficulty: basic 90, intermediate 120, advanced 160 (minutes). The clamp ensures \(q_{\text{time}}\in[0.6,1.0]\).
Event contribution (coding — Micro/Interview)
$$\text{contrib}_{\text{code}}(e)=w_{a(e)}\cdot o(e)\cdot f_{d(e)}\cdot \text{decay}(e)$$

Read as: (track importance) × (verified quality) × (difficulty) × (recency). This applies only to accepted coding solves in the micro and interview tracks.

Event contribution (MCQ quiz result)
$$\text{contrib}_{\text{mcq}}(e)=\left(\frac{p(e)}{100}\right)^{\gamma}\cdot f_{d(e)}\cdot \text{decay}(e)$$

Here \(p(e)\in[0,100]\) is the quiz percentage mark (it may reflect partial completion). MCQ does not use \(o(e)\), attempts buckets, or coding time penalties. Default \(\gamma=1.5\) so higher marks contribute disproportionately more evidence than low marks.

Evidence per track and category
$$E_{a,c}=\sum_{e\in\mathcal{E}_a:\,c(e)=c}\text{contrib}_a(e)$$

Where \(\text{contrib}_a\) is \(\text{contrib}_{\text{code}}\) for \(a\in\{\text{micro},\text{interview}\}\), and \(\text{contrib}_{\text{mcq}}\) for \(a=\text{mcq}\). Breadth is computed per track from these values. Depth for mastery is computed for the two coding tracks separately (no cross-track pooling).

Step 2 — Breadth (per track)
Breadth asks “how wide is your verified competence?”

Why breadth is not just “how many categories you touched”

Breadth has three parts: (1) coverage (how many categories have meaningful evidence), (2) balance (how evenly your evidence is distributed), and (3) confidence (prevents early users from getting unstable scores from tiny data).

We compute breadth separately per track so the dashboard can show whether you’re broad in Micro, broad in Interview, and broad in MCQ.

Activation threshold (a category counts as covered if evidence is real)
$$A_{a,c}=\mathbf{1}\{E_{a,c}\ge\tau_E\}$$

\(\tau_E\) prevents “one tiny action” from marking a category as covered. Default \(\tau_E=1.5\).

Coverage (fraction of covered categories)
$$\text{cov}_a=\frac{1}{|C_a|}\sum_{c\in C_a}A_{a,c}$$

This is the simplest breadth idea: what fraction of the category set has meaningful evidence.

Balance (entropy of evidence distribution, normalized)
$$p_{a,c}=\frac{E_{a,c}}{\sum_{j\in C_a}E_{a,j}}$$ $$H_a=-\sum_{c\in C_a}p_{a,c}\ln p_{a,c}$$ $$\text{bal}_a=\frac{H_a}{\ln |C_a|}$$

Balance prevents “breadth inflation” by concentrating all evidence in one category. If evidence is spread evenly, entropy is higher. Edge case: if total evidence \(\sum_{j\in C_a}E_{a,j}=0\), we define \(H_a=0\) and \(\text{bal}_a=0\).

Confidence stabilizer (prevents early volatility)
$$T_a=\sum_{c\in C_a}E_{a,c}$$ $$\text{conf}_a = 1-e^{-T_a/k_a}$$

When evidence is tiny, confidence is small, and breadth stays conservative. As evidence accumulates, confidence approaches 1 and breadth reflects coverage + balance. Defaults: \(k_{\text{micro}}=10\), \(k_{\text{interview}}=8\), \(k_{\text{mcq}}=14\).

Breadth per track (0–100)
$$B_a = 100\cdot \text{conf}_a \cdot \left(\alpha\cdot \text{cov}_a+\beta\cdot \text{bal}_a\right)$$

We blend coverage and balance. Defaults: \(\alpha=0.65\) and \(\beta=0.35\). This puts slightly more weight on “cover more categories” while still penalizing narrow concentration. Edge case: if \(T_a=\sum_{c\in C_a}E_{a,c}=0\), then \(\text{cov}_a=0\), \(\text{bal}_a=0\), \(\text{conf}_a=0\), and therefore \(B_a=0\).

Overall breadth across tracks (0–100)
$$B=0.4\,B_{\text{micro}}+0.4\,B_{\text{interview}}+0.2\,B_{\text{mcq}}$$

These fixed weights make the headline score predictable and transparent: coding breadth dominates, while MCQ breadth still matters. If a track has zero evidence then its breadth is 0 by definition (see edge cases below).

Step 3 — Depth (coding tracks)
Depth asks “how strong are you within each category?” (computed per coding track; no cross-track pooling).

Why depth uses a saturating curve

Depth should have diminishing returns: the difference between “0 and some evidence” is huge, but the difference between “a lot and a little more” should not explode the score. We therefore map evidence into a 0–100 depth value per category using a saturating curve.

Categories are not pooled across tracks because each track has its own category set. Depth for mastery is defined for the two coding tracks (micro + interview) and then combined. MCQ contributes to breadth and points, but does not contribute to mastery depth in the initial version.

Depth per category (per coding track, 0–100, saturating)
$$D_{a,c}=100\left(1-e^{-E_{a,c}/\lambda}\right)\qquad a\in\{\text{micro},\text{interview}\}$$

\(\lambda\) controls how quickly depth rises. Default \(\lambda=12\). At \(E_{a,c}=\lambda\), depth reaches about 63% of its maximum.

Track depth (0–100, evidence-weighted)
$$D_a= \begin{cases} \dfrac{\sum_{c\in C_a}E_{a,c}D_{a,c}}{\sum_{c\in C_a}E_{a,c}}, & \text{if }\sum_{c\in C_a}E_{a,c}>0\\[8pt] 0, & \text{otherwise} \end{cases}$$

We weight by evidence so categories with real work influence depth more than categories with tiny evidence. Edge case: if a track has zero evidence, its depth is defined as 0.

Overall depth used in CodiviumScore (0–100)
$$D = 0.5\,D_{\text{micro}} + 0.5\,D_{\text{interview}}$$

Depth is intentionally coding-centric in the initial version, matching the depth-by-category chart (coding categories only).

Depth-by-category bar chart (coding categories union; chart-only)
$$E^{\text{chart}}_{c}=E_{\text{micro},c}+E_{\text{interview},c}\quad\text{(merged by canonical category key)}$$ $$D^{\text{chart}}_{c}=100\left(1-e^{-E^{\text{chart}}_{c}/\lambda_{\text{chart}}}\right)$$

The chart uses the union of coding categories across micro + interview. If the same canonical category key exists in both tracks, evidence is summed before applying the saturating curve. Default \(\lambda_{\text{chart}}=12\). This chart is a visualization of coding depth by category and does not introduce cross-track pooling into the mastery score.

Step 4 — Codivium Score (Mastery)
The headline score is capped 0–100 and punishes imbalance.

Why the harmonic mean is used

Elite developers are both broad and deep. The harmonic mean enforces that: if either breadth or depth is weak, the combined score is pulled down. This matches deliberate practice: you cannot claim “elite” if you avoid large parts of the skill surface.

Codivium Score (0–100)
$$\text{CodiviumScore}=\frac{2BD}{B+D}$$

Both \(B\) and \(D\) are in \([0,100]\), so the final score is also in \([0,100]\). Edge case: if \(B=0\) or \(D=0\), we define \(\text{CodiviumScore}=0\). This makes the headline score stable and interpretable.

Supporting Metric — Efficiency (pts/hr)
Uncapped, hard to game: only accepted solves count; slow solves reduce the rate.

Design goal: reward time only when it produces verified outcomes

Efficiency exists because two users can have similar mastery levels but very different practice quality: one may produce accepted solutions quickly and consistently, while another may spend the same time on fewer accepted solves.

Efficiency therefore behaves like a “throughput” metric: it increases with more accepted solves, and it increases when accepted solves are achieved with less time. Because only accepted solves are included, it does not reward unproductive time.

Recency weights for efficiency (so recent performance dominates)
$$w_i=0.5^{\Delta_i/H}$$

\(\Delta_i\) is days since solve \(i\), and \(H\) is the same half-life as used in evidence (default 15 days).

Solve quality weight for efficiency (premium)
$$q_i = f_{d(i)}\cdot q_{\text{tries}}(i)\cdot q_{\text{conv}}(i)$$

For efficiency we exclude the explicit time penalty inside \(q_i\) because time is already accounted for in the denominator. This avoids “double counting” time. Difficulty, tries, and convergence remain because they reflect quality.

Efficiency (uncapped) and displayed as pts/hr
$$\text{Efficiency}=\frac{\sum_i w_iq_i}{\epsilon+\sum_i w_it_i}$$ $$\text{Efficiency(pts/hr)} = 60\cdot \text{Efficiency}$$

\(\epsilon\) is a small constant (default 1 minute) to avoid division by zero. \(t_i\) is time-to-AC in minutes. The score is uncapped: as a user gets faster and completes more accepted solves, efficiency can continue to rise.

Supporting Metric — Codivium Points (uncapped)
Progression: bank verified work over a career, and track 30-day momentum.

Design goal: reward verified volume without inflating mastery

Points exist because a 0–100 mastery score is intentionally capped. Elite users still need a way to see long-term progression: “How much verified work have I accumulated?” and “How much did I progress recently?”

Points are uncapped but carefully designed: they reward distinct achievements more than repeats, and repeats have diminishing returns and spacing rules to prevent farming.

Parameters (defaults)

Default points parameters: P0=4.0 (first accepted solve), Predo=2.5 (redo baseline), Pmcq=0.8 (MCQ all-time padding via \(\sqrt{U_{\text{all}}}\)), G=7 days (minimum spacing between counted re-dos).

Distinct coding points (first accepted solve per ExerciseId)
$$S_{\text{first}}=\{\text{firstAcceptedSolve}(x)\ \forall\ \text{ExerciseId }x\}$$ $$\text{CodingPoints}_{\text{all}}=\sum_{e\in S_{\text{first}}}P_0\cdot w_{a(e)}\cdot f_{d(e)}\cdot q_{\text{conv}}(e)$$ $$\text{CodingPoints}_{30}=\sum_{e\in S_{\text{first}}\cap W_{30}}P_0\cdot w_{a(e)}\cdot f_{d(e)}\cdot q_{\text{conv}}(e)$$

Why this works:

  • Distinct means you get the main credit once per exercise (requires ExerciseId).
  • We still reward difficulty and convergence so that “hard clean wins” bank more points.
  • The 30-day version measures momentum: how much distinct ground you covered recently.
Redo points (accepted re-solves, spaced, diminishing returns)
$$\text{validRedo}(x,k)=\mathbf{1}\{t_k(x)-t_{k-1}(x)\ge G\text{ days}\}$$ $$R_{\text{valid,all}}(x)=\sum_{k\ge2}\text{validRedo}(x,k)$$ $$R_{\text{valid,30}}(x)=\sum_{k\ge2}\text{validRedo}(x,k)\cdot \mathbf{1}\{t_k(x)\in W_{30}\}$$ $$\text{RedoPoints}_{\text{all}}=\sum_x P_{\text{redo}}\cdot w_{a(x)}\cdot f_{d(x)}\cdot \ln(1+R_{\text{valid,all}}(x))$$ $$\text{RedoPoints}_{30}=\sum_x P_{\text{redo}}\cdot w_{a(x)}\cdot f_{d(x)}\cdot \ln(1+R_{\text{valid,30}}(x))$$

The spacing rule \(G\) discourages immediate repeats. The log term makes the reward sublinear: repeating the same exercise helps (spaced repetition matters), but you cannot inflate points linearly by farming one item. Here \(a(x)\) and \(d(x)\) are the track and difficulty of exercise \(x\) (exercise metadata; assumed stable across attempts).

MCQ points (all-time only, unique correct count with sqrt)
$$\text{McqPoints}_{\text{all}}=P_{\text{mcq}}\sqrt{U_{\text{all}}}$$ $$\text{McqPoints}_{30}=0\quad\text{(initial implementation)}$$

MCQ points are a separate uncapped progression signal and are not used in the mastery score. They use \(U_{\text{all}}\): lifetime distinct MCQ questions answered correctly. MCQ breadth, by contrast, is computed from quiz results (category + difficulty + percent mark) with the same recency decay as coding. The sqrt makes early progress feel rewarding while keeping long-term growth controlled.

Final points totals
$$\text{Points}_{\text{all}}=\text{CodingPoints}_{\text{all}}+\text{RedoPoints}_{\text{all}}+\text{McqPoints}_{\text{all}}$$ $$\text{Points}_{30}=\text{CodingPoints}_{30}+\text{RedoPoints}_{30}$$

These are the two point values shown on the dashboard: All-time (career bank) and last-30-days (momentum).

Progress Panel, Primary Driver, and CTAs
How the dashboard turns analytics into deliberate practice.

Why progress needs deltas, not just totals

Premium dashboards do not only show “where you are”; they show “how you are moving”. The Progress Panel therefore displays: (a) recent deltas (e.g., last 7 days), (b) small trend lines (sparklines), (c) a primary driver explanation, and (d) one next action.

These are computed in the backend so the explanation is consistent with the score math, and so it can be tuned via config.

Example delta computations (backend)
$$\Delta\text{Score}_{7d}=\text{CodiviumScore}(\text{today})-\text{CodiviumScore}(\text{today}-7\text{ days})$$ $$\Delta\text{Points}_{30}=\text{Points}_{30}(\text{today})-\text{Points}_{30}(\text{today}-7\text{ days})$$

The backend stores (or recomputes) daily values so the UI can show deltas and trend lines without duplicating math in the browser.

Primary driver (config-driven choice)

The backend selects a “primary driver” mode from config. Example modes:

  • Mastery decomposition (default): did breadth or depth contribute more to \(\Delta\)Score?
  • Largest category evidence gain: which category increased \(E_c\) most in the selected window?
  • Efficiency improvement: which factor increased pts/hr most?
  • Heatmap improvement: which focus categories improved early convergence (A1/A2) most?

The chosen mode produces: a short title, a one-paragraph explanation, and a few supporting numbers for transparency.

Purpose of CTA buttons (how the dashboard becomes actionable)

CTA buttons exist to close the deliberate practice loop:

  • Analytics → Prescription: the backend computes recommended actions from the same aggregates used in the charts.
  • Prescription → Session: clicking a CTA calls a backend endpoint that creates a practice session and returns a redirect URL.
  • Session → Evidence: completing the session generates verified events, increasing evidence and therefore improving breadth/depth/score.

This is the “elite” product behavior: the dashboard does not merely report; it directs.

Step 5 — Convergence Heatmap
How the heatmap cell values and focus-mode rankings are computed.

What the heatmap measures

The convergence heatmap shows, for each coding category and each attempt number, how correct your code was on average at that exact submission. This is different from a cumulative pass rate — it measures quality at attempt k, not "whether you eventually passed by attempt k."

This is useful because it reveals how quickly your code converges to correct: a category with high A1 values means you typically submit near-correct solutions immediately; low A1 but rising A3/A4 means you need several iterations to converge.

Cell value — mean unit-test pass ratio at attempt k

For a session with N unit tests, the ratio at attempt k is:

$$r_k(s) = \text{clamp}\!\left(\frac{\text{UnitTestsPassed}_k(s)}{N},\;0,\;1 ight)$$

The cell value for category c at bucket A_k is the mean across all sessions in category c that had at least k attempts:

$$\text{cell}(c,\,k) = \begin{cases} \dfrac{1}{|\mathcal{S}_{c,k}|}\displaystyle\sum_{s \in \mathcal{S}_{c,k}} r_k(s) & \text{if } |\mathcal{S}_{c,k}| > 0 \[8pt] \text{null} & \text{otherwise} \end{cases}$$

where \(\mathcal{S}_{c,k}\) is the set of sessions in category \(c\) that had at least \(k\) submission attempts. Sessions with fewer than \(k\) attempts contribute nothing to bucket \(A_k\) — they are absent, not counted as zero. A null is displayed when no session reached that attempt count (rather than zero, which would be misleading). Buckets: A1..A5 correspond to attempt indices 1..5. Micro and interview sessions are pooled for the combined view.

Why the cell value is a mean ratio, not a cumulative pass fraction

A cumulative pass fraction would ask "what fraction of users passed by attempt k?" The mean unit-test ratio instead asks "how correct was the code at attempt k on average?" This is more informative for deliberate practice: it shows whether partial progress is being made (rising ratio across buckets) or whether submissions are random (flat ratio).

Focus modes — what the dropdown controls

The heatmap dropdown selects a focus mode: a ranking of categories from most to least worth practising under that lens. All modes use the same underlying cell values; they differ only in how they rank categories.

Shared derived inputs used by all modes:

Early convergence (input to focus mode rankings)
$$\text{earlyConv}(c) = \frac{1}{|\{k : \text{cell}(c,k) e \text{null},\, k \le 3\}|} \sum_{\substack{k=1 \ \text{cell}(c,k) etext{null}}}^{3} \text{cell}(c,k)$$

Mean of the A1, A2, A3 cell values for category \(c\), averaging only the buckets that have data. If no sessions reached A1–A3, \(\text{earlyConv}(c)=0\).

$$\text{weakness}(c) = \text{clamp}(1 - \text{earlyConv}(c),\;0,\;1)$$ $$\text{depthNorm}(c) = \text{clamp}\!\left(\frac{D^{\text{chart}}_c}{100},\;0,\;1 ight)$$ $$\text{timeWeight}(c) = \ln\!\left(1 + \max(0,\,\text{minutes}_c) ight)$$

where \(\text{minutes}_c\) is the total allocation minutes for category \(c\) and \(D^{\text{chart}}_c\) is the depth-by-category chart value from Step 3. The log scaling compresses large time differences so very-high-time categories do not dominate rankings purely due to accumulated time.

Focus mode: Impact (default)
$$\text{score}_{\text{impact}}(c) = \bigl(alpha_{\text{imp}}\cdottext{weakness}(c) + beta_{\text{imp}}\cdot(1-\text{depthNorm}(c))\bigr) \times \text{timeWeight}(c)$$

Prioritises categories where early-attempt quality is low and depth is weak, scaled by how much time you have already invested there. Categories where you spend time but still submit poor code and have low depth rank highest. Default blend weights: αimp=0.55, βimp=0.45 (stored in config under heatmap.focus_modes.impact_weights). Ties broken alphabetically by category name.

Focus mode: Worst Convergence
$$\text{score}_{\text{worst\_conv}}(c) = \text{weakness}(c)$$

Ranks categories by how low early-attempt code quality is, descending. Higher weakness = more wrong answers on first attempts = ranked higher. Ties broken alphabetically.

Focus mode: Most Time Spent
$$\text{score}_{\text{most\_time}}(c) = \text{minutes}_c$$

Ranks by raw minutes allocated to the category, descending. Shows where practice time is currently concentrated. Ties broken alphabetically.

Focus mode: Lowest Depth
$$\text{score}_{\text{lowest\_depth}}(c) = 1 - \text{depthNorm}(c)$$

Ranks by weakest depth score, descending (inverted so lowest depth ranks first). Ties broken alphabetically.

Focus mode: Highest Friction
$$\text{score}_{\text{friction}}(c) = \text{weakness}(c) \times \text{timeWeight}(c)$$

Ranks categories where you spend significant time but still produce low-quality early submissions. Similar to Impact but without the depth term — pure "effort × poor early quality." Ties broken alphabetically.

Step 6 — Supplementary Diagnostics
Dashboard metrics that are not part of the mastery score but provide practice-quality context.

What supplementary diagnostics are and why they exist

The mastery score (Steps 1–4) answers "how strong are you right now?" Supplementary diagnostics answer "how is your practice process performing?" They are computed from the same raw session data but are not inputs to CodiviumScore, Breadth, Depth, or Points.

All supplementary diagnostics use accepted coding sessions only (sessions where at least one attempt passed all unit tests), combining micro and interview tracks.

First-try pass rate (%)
$$\text{FirstTryPass} = 100 \cdot \frac{\bigl|\{s \in \mathcal{S}_{\text{accepted}} : \text{attemptsToAC}(s) = 1\}\bigr|} {|\mathcal{S}_{\text{accepted}}|}$$

Percentage of accepted sessions where the very first submission passed all unit tests. Measures precision and pre-submission planning quality. \(\text{attemptsToAC}(s)\) = index of the last accepted attempt + 1 (total submissions in the session up to and including the final pass). Returns 0 if \(|\mathcal{S}_{\text{accepted}}| = 0\).

Average attempts to accepted
$$\text{AvgAttempts} = \frac{1}{|\mathcal{S}_{\text{accepted}}|} \sum_{s \in \mathcal{S}_{\text{accepted}}} \text{attemptsToAC}(s)$$

Mean number of submissions required to reach a passing solution across all accepted sessions. Returns 0 if \(|\mathcal{S}_{\text{accepted}}| = 0\).

Median time to accepted (minutes)
$$\text{MedianTimeToAC} = \text{median}\!\left(\{t_{\text{AC}}(s) : s \in \mathcal{S}_{\text{accepted}}\} ight)$$

Median of \(t_{\text{AC}}(s)\) across accepted sessions, where \(t_{\text{AC}}(s) = \text{TimeToACMinutes}(s)\) if explicitly provided, otherwise the TimeToSubmissionMinutes of the last accepted attempt. The median is used rather than the mean to avoid skew from rare very-long sessions. Returns 0 if \(|\mathcal{S}_{\text{accepted}}| = 0\).

Weighted MCQ score (%)

A difficulty-weighted quality indicator for MCQ performance. For each difficulty level \(d \in \{\text{basic, intermediate, advanced}\}\):

$$\bar{p}_d = \frac{1}{|\mathcal{Q}_d|}\sum_{q \in \mathcal{Q}_d} \text{ScorePercent}(q) \qquad \text{(mean score at difficulty } dtext{)}$$ $$\text{WeightedMcqScore} = \frac{\displaystyle\sum_{d} f_d \cdot \bar{p}_d} {\displaystyle\sum_{d} f_d}$$

where \(f_d\) is the standard difficulty factor (basic=1.0, intermediate=1.3, advanced=1.6) and \(\mathcal{Q}_d\) is the set of all MCQ quiz attempts at difficulty \(d\). If a difficulty level has no attempts, it is excluded from both numerator and denominator. Result is in \([0, 100]\).

Time on platform (daily series)

Daily minutes (time-on-platform series)
$$\text{minutes}(d) = \sum_{\substack{s \in \mathcal{S}_{\text{accepted}} \ \text{date}(s) = d}} t_{\text{AC}}(s) \;+\; \sum_{\substack{q \in \mathcal{Q}_{\text{all}} \ \text{date}(q) = d}} \text{MinutesSpent}(q)$$

Total minutes on a given calendar day, combining: (1) TimeToACMinutes from accepted coding sessions whose CompletedAtUtc falls on day \(d\), and (2) MinutesSpent from MCQ quiz attempts on day \(d\) (defaulting to a 6-minute estimate per attempt if not provided). Days with no activity are omitted from the series. The series is bounded by the query window (from/to parameters).

Allocation by category

Category allocation (minutes and solved count)
$$\text{minutes}(c) = \sum_{\substack{s \in \mathcal{S}_{\text{accepted}} \ \text{category}(s) = c}} t_{\text{AC}}(s)$$ $$\text{solved}(c) = \bigl|\{s \in \mathcal{S}_{\text{accepted}} : \text{category}(s) = c\}\bigr|$$

Total minutes spent on accepted sessions in category \(c\), and the count of those sessions. Combines micro and interview tracks. Categories with zero accepted sessions are omitted. Used for the time-by-category allocation chart.

MCQ performance charts

MCQ score by category and difficulty
$$\text{McqScore}(c,\,d) = \begin{cases} \dfrac{1}{|\mathcal{Q}_{c,d}|}\displaystyle\sum_{q \in \mathcal{Q}_{c,d}} \text{ScorePercent}(q) & \text{if } |\mathcal{Q}_{c,d}| > 0 \[8pt] \text{null} & \text{otherwise} \end{cases}$$

Arithmetic mean of ScorePercent across all MCQ quiz attempts in category \(c\) at difficulty \(d\). \(\mathcal{Q}_{c,d}\) is the set of all MCQ attempts in that category/difficulty cell. A null is shown when no attempts exist for that cell (not zero). Evidence fields — attempt count and total questions — are included alongside the score value for tooltip context.

Configuration Defaults (Source of Truth)
All defaults, formulas, and clarifying decisions are documented here and mirrored in config.

Why defaults live in both docs and config

This document is the human-readable scoring reference. The same defaults must also exist in a machine-readable config (backend config file or equivalent) so the implementation stays aligned over time. Any future tuning must update both the config and this document.

Track decisions (documented)
  • Tracks: micro (coding), interview (coding), mcq.
  • Each track has its own category set; categories are not pooled across tracks for mastery computations.
  • MCQ contributes to breadth (via quiz results) and to points (via \(U_{\text{all}}\)); MCQ does not contribute to mastery depth in the initial version.
  • Depth-by-category chart uses the union of coding categories (micro + interview), merged by canonical category key by summing evidence.
  • Standard edge-case rule: any function depending on total evidence returns 0 when total evidence is 0.
Core constants
  • Recency half-life: H=15 days.
  • Breadth activation threshold: τE=1.5.
  • Breadth blend: α=0.65, β=0.35.
  • Confidence stabilizers: k_micro=10, k_interview=8, k_mcq=14.
  • Depth saturating constant: λ=12 (and λ_chart=12 for the depth-by-category chart).
Difficulty mapping (all tracks)

Codivium difficulties are: basic, intermediate, advanced. Map to numeric factors:

  • basic → 1.0
  • intermediate → 1.3
  • advanced → 1.6
Coding quality defaults
  • tries factor clamp: min=0.45, max=1.0
  • convergence weights: g=[1.00, 0.92, 0.85, 0.78, 0.70, 0.60]
  • attempt bucket: A=min(attempts, 6) (A1..A5 for 1..5 attempts, A6 for >5)
  • time factor clamp: min=0.6, max=1.0
  • time penalty constants \(T_d\) (minutes): basic 90, intermediate 120, advanced 160
  • activity weights for coding evidence/points: w_micro=1.0, w_interview=1.0
MCQ breadth defaults
  • Recency decay: same \(H\) as coding.
  • Evidence from quiz marks: exponent γ=1.5 in \( (p/100)^\gamma \).
  • MCQ quizzes have a single difficulty level (basic/intermediate/advanced) per quiz.
Aggregation into headline mastery score
  • Overall breadth: \(B=0.4B_{micro}+0.4B_{interview}+0.2B_{mcq}\).
  • Overall depth: \(D=0.5D_{micro}+0.5D_{interview}\).
  • Headline score: \(\text{CodiviumScore}=\frac{2BD}{B+D}\) with \(\text{CodiviumScore}=0\) if \(B=0\) or \(D=0\).
Points + efficiency defaults
  • First-solve points constant: P0=4.0
  • Redo points constant: Predo=2.5
  • Redo spacing: G=7 days
  • MCQ points scaling: Pmcq=0.8 in \(P_{mcq}\sqrt{U_{all}}\)
  • Efficiency stabilizer: ε=1.0 minute
Canonical category key (for chart merging)

Default canonical key rule: use lower(trim(CategoryId)) if CategoryId is stable across coding tracks; otherwise use lower(trim(CategoryName)). Maintain a small alias mapping table in config for known merges (optional).

Machine-readable defaults (JSON)

Defaults are stored in a separate config file for easy backend integration: scoring-config.katex.v1.json.
