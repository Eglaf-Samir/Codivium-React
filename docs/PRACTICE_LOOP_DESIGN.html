Section 1

The loop at a glance

The practice loop is the core activity cycle of Codivium. It has three stages:

Menu Browse all exercises for a track. Filter by category, difficulty, completion. Search by name. Click a card to enter the loop.
↓
Editor Work on a single exercise. Read the problem, write code, submit, see test results. Iterate freely. No interruption on failure.
↓ (on success only)
Feedback Floating modal over the editor. Immediate performance analysis. Three explicit choices: Dashboard, Return to Menu, Try Again.
↓
Back to loop User chooses where to go. Loop continues or breaks. New session data feeds the dashboard scoring model.

The loop is intentionally asymmetric. Success triggers a ceremony — the feedback modal appears, progress is acknowledged, and a clear decision is required before moving on. Failure is silent — the editor remains open and the user retains full control with no interruption. This asymmetry is a deliberate design choice, not an oversight.

Section 2

Exercise Menu

File: menu-page.html. One HTML file serves both the Micro Challenges and Interview Preparation tracks via a URL parameter: menu-page.html?track=micro or menu-page.html?track=interview. The page reads the track from the URL on load, fetches the appropriate exercise list, and renders everything from that data. The MCQ track does not use this page — it has its own separate flow.

Data payload

On load the page calls GET /api/menu/payload?track=micro (or interview). The server returns a single JSON object:

{
  "track": "micro",
  "trackLabel": "Micro Challenges",
  "categories": ["Arrays", "Strings", "Trees", /* 14 total */],
  "difficultyLevels": ["basic", "intermediate", "advanced"],
  "exercises": [
    {
      "id": "micro_042",
      "name": "Minimum Window Substring",
      "shortDescription": "Find the smallest substring containing all required characters.",
      "category": "Strings",
      "difficulty": "advanced",
      "completionStatus": "completed",
      "completedAt": "2026-02-14"
    }
    /* ... all exercises for this track */
  ]
}

The completionStatus field uses three values: "completed" (at least one accepted solve), "attempted" (submitted but never passed all tests), "not_started" (never opened). The current filter drawer UI shows only completed / not-started, but the third state is carried in the payload so it can be added to the filter later without a schema change.

The categories and difficultyLevels arrays are sent explicitly (not derived from the exercises list) so the filter drawer always shows all options even if no exercises currently exist in a given category or difficulty.

All filtering and sorting happens entirely on the client side in memory. The full list is delivered once and never re-fetched during a browse session. At 200–1,000 exercises this is fast, the payload is small (under 100KB gzipped), and there is no round-trip cost on every filter interaction.

Card structure

Each exercise is rendered as a card in a 6-column CSS grid. The card HTML generated from the payload looks like this:

<a class="exercise-card"
   data-cat="strings"
   data-level="advanced"
   data-status="completed"
   data-id="micro_042"
   href="editor.html?id=micro_042&from=menu-page.html%3Ftrack%3Dmicro">

  <div class="card-status done">...tick svg...</div>
  <div class="card-body">
    <h3 class="card-title">Minimum Window Substring</h3>
    <div class="card-desc">Find the smallest substring...</div>
  </div>
  <div class="pill-row">
    <span class="pill">Strings</span>
    <span class="pill pill-level advanced">Advanced</span>
  </div>
</a>

Four data-* attributes drive the filter logic: data-cat, data-level, data-status, data-id. All values are stored in lowercase. The card-status element uses class done (green tick), todo (yellow cross), or attempted (grey partial) to show completion state visually. The card is an <a> element so clicking navigates directly — no JS click handler needed for the navigation itself.

Filtering and sorting

The filter logic in drawer-and-filters.js is already implemented and working against the current static cards. It reads three sets of checkboxes (category, difficulty, completeness), reads the sort key and direction from two dropdowns, and applies them to the card DOM nodes in the grid. The filter function runs when the Apply button is clicked.

When the page is wired to the real data payload, the only change needed in the filter JS is:

  1. The filter drawer's category checkboxes need to be dynamically generated from payload.categories instead of being hardcoded in the HTML. Currently they are hardcoded to a subset of categories.
  2. The getCardMeta() function reads data-cat, data-level, and card-status.classList from each card. These are already the right attribute names — the dynamic card renderer just needs to write them correctly.
  3. The sort comparators already handle title, category, level, and completeness. No changes needed there.

The filter operates by showing/hiding cards via style.display. Cards that do not match the active filters are hidden; matching cards are re-ordered in the DOM via appendChild into the grid in sorted order. This is fast at the expected scale.

Search

A text input field sits above the exercise grid (outside the collapsible filter drawer, always visible). As the user types, the exercise list filters in real time. The search matches against data-name and data-desc attributes on each card using case-insensitive substring matching. The search result is combined with the filter drawer's active selections via AND — a card must satisfy both the text search and any active filters to be shown.

This field does not exist in the current HTML yet. It needs to be added — a single <input> above the grid, with a corresponding event listener in the filter JS. The logic is a small addition to the existing applyFilters() function.

Navigating to an exercise

Clicking any card navigates to the editor page. The URL carries two pieces of information:

editor.html?id=micro_042&from=menu-page.html%3Ftrack%3Dmicro

id is the globally unique exercise ID. from is the full return URL, URL-encoded, so the feedback modal's "Return to Menu" button sends the learner back to the exact same page and track they came from. The browser back button also works naturally. No session state or localStorage is needed.

Exercise ID uniqueness

If the micro challenges and interview exercises are in separate database tables with independent ID sequences, their numeric IDs may collide. The scoring engine uses ExerciseId for distinct-solve counting and redo-spacing — collisions corrupt both. Prefix with the track: micro_42 and interview_42 for exercise #42 in each track respectively.

Section 3

Editor Page

File: editor.html (not yet built). The editor is the coding environment for a single exercise. It follows the same shared component structure as all other pages — the same topbar, the same sidebar with the same buttons, the same logo and cube.

Loading an exercise

On load the page reads ?id= from the URL and calls:

GET /api/exercise?id=micro_042

The response contains everything needed to render the exercise view:

{
  "id": "micro_042",
  "name": "Minimum Window Substring",
  "category": "Strings",
  "difficulty": "advanced",
  "problemStatement": "Given a string s and a string t, find...",
  "constraints": ["1 <= s.length <= 10^5", /* ... */],
  "examples": [{ "input": "s=ADOBECODEBANC,t=ABC", "output": "BANC" }],
  "codeScaffold": "def minWindow(s: str, t: str) -> str:\n    pass",
  "testsTotal": 24,
  "priorAttempts": 3,
  "bestPriorScore": 92,
  "lastSolvedAt": "2026-02-14"
}

The priorAttempts, bestPriorScore, and lastSolvedAt fields allow the editor to show the learner their history with this exercise before they start — "You have solved this before. Your best previous run: 3 attempts, 92% peak." This is optional UI chrome, not required for the editor to function.

Submitting code

When the learner clicks Submit, the editor POSTs their code to:

POST /api/exercise/submit
{
  "exerciseId": "micro_042",
  "code": "def minWindow(s, t):\n    ...",
  "sessionId": "sess_abc123"   // session created when the editor was first opened
}

The server runs the test suite against the submitted code and returns a result. The response has two shapes depending on the outcome.

Failed submission response

{
  "accepted": false,
  "testsPassed": 8,
  "testsTotal": 24,
  "failedTests": [
    { "input": "...", "expected": "...", "got": "..." }
  ],
  "timeMs": 42
}

Successful submission response

{
  "accepted": true,
  "testsPassed": 24,
  "testsTotal": 24,
  "feedback": {
    "exerciseId": "micro_042",
    "exerciseName": "Minimum Window Substring",
    "category": "Strings",
    "difficulty": "advanced",
    "isFirstSolve": false,
    "returnMenuUrl": "menu-page.html?track=micro",
    "current": {
      "timeToSuccessSeconds": 768,
      "attempts": 2,
      "bestPreSuccessPercent": 92,
      "timeToFirstAttemptSecs": 125,
      "avgIterationSeconds": 314
    },
    "history": [
      { "timeToSuccessSeconds": 1020, "attempts": 4 },
      /* previous completions, oldest first, up to 8 */
    ],
    "deltas": {
      "timeToSuccess": -11,   // % change vs most recent prior completion
      "attempts": -33
    }
  }
}

Notice the feedback payload is nested inside the submission response. This means one round trip handles both "did I pass?" and "here is your analysis." The editor does not need a separate call to get the feedback data — it arrives with the acceptance confirmation.

On failure — no dialog

When accepted: false is returned, the editor updates the test results panel to show which tests passed and which failed, with input/expected/got detail for failed tests. Nothing else happens. No modal, no dialog, no sound, no interruption. The code editor remains focused. The learner decides what to do next entirely on their own terms — fix the code and resubmit, open the sidebar to navigate away, or press the browser back button to return to the menu.

The sidebar is always present on the editor page. The learner can navigate away at any point, including mid-exercise, using any sidebar button. This is the same sidebar as every other page — same buttons, same positions, same behaviour.

On success — show the feedback modal

When accepted: true is returned, the editor extracts the feedback object from the response and calls:

CodiviumFeedback.show(result.feedback);

This is the entire integration with the feedback component. The modal appears as a floating overlay. The editor and the learner's accepted code remain fully visible behind it. The learner sees their solution and the feedback simultaneously.

The returnMenuUrl field in the feedback payload is constructed server-side from the from query parameter that arrived in the original editor URL. The server reads ?from=, decodes it, and echoes it into the feedback payload so the "Return to Menu" button navigates to exactly the right place.

Section 4

Feedback Modal

Files: assets/components/feedback/feedback.css, assets/components/feedback/feedback.js. Fully built. The public API is a single function: CodiviumFeedback.show(result). The modal is added to the DOM programmatically — it is not present in any page's HTML. It appears as a floating 1040px-wide overlay on top of whatever page calls it, with a blurred semi-transparent backdrop that keeps the editor visible behind it.

Structure and content

The modal has a header, a four-row body, and a footer.

Header — Green checkmark seal, exercise name, three pills (category, difficulty, test count). No close button. The only way out is the three footer buttons.

Row 1: Primary metrics + Insight — Left side: Time to Success (mm:ss) and Attempts (count), each with a coloured delta badge (green down-arrow for improvement, red up-arrow for regression, compared to the most recent prior completion of this exercise). Right side: the Insight panel — a gold-bordered block with analysis text and badge chips.

Row 2: Secondary micro-metrics — Three smaller cards: Time to First Attempt (how long before the first submission — a proxy for planning vs diving in), Best Pre-Success pass rate (the highest test percentage achieved before the final passing submission — shows how close the initial solution was), and Average Iteration time (mean time between consecutive submissions — measures the efficiency of the fix-and-retry loop).

Row 3: Trend charts — Only rendered when history contains 2 or more prior completions. Two inline SVG charts side by side: Time to Success Trend (line chart with area fill and dot markers, last 8 completions) and Attempts Trend (bar chart with overlay trend line, last 8 completions). Both show Best / Median / Latest in a text legend below.

Footer — "Codivium — Immediate feedback" stamp on the left. Three buttons on the right.

Rule-based insight engine

The insight text is generated synchronously by a rule set inside feedback.js. It evaluates the current run and produces a 2–3 sentence personalised paragraph. Rules in order of evaluation:

Convergence quality rule — based on attempt count:

  • 1 attempt → "First-attempt acceptance — your pre-submission reasoning was precise."
  • 2 attempts → references the best pre-success percent: "Your first attempt reached X% of tests — one focused iteration closed the gap."
  • 3–4 attempts → "The failure pattern was identified and resolved cleanly."
  • 5+ attempts → "Consider writing a small concrete test case before submitting — it tends to surface the edge case responsible for most retries."

Time quality rule — compares timeToSuccessSeconds against the KaTeX T_d benchmark for this difficulty (basic = 5,400s / 90min, intermediate = 7,200s / 120min, advanced = 9,600s / 160min):

  • Under 30% of T_d → "Well inside the reference window — strong signal of a solid mental model."
  • 30–60% → "Comfortably within the expected range."
  • 60–100% → "Approached the reference window — a written plan before coding is usually faster overall."
  • Over 100% → Recommends timeboxing and post-mortem.

Historical trend rule — if prior history exists:

  • Time improved by more than 10% → explicitly calls it out: "X% faster than your previous attempt."
  • Attempts reduced by more than 15% → adds a "Fewer attempts" chip.
  • No prior history → adds a "First solve" chip.

Badge chips are assembled alongside the text and shown in the insight panel as small pill elements. They summarise the key signals at a glance: "First-try pass", "Fast solve", "Personal best time", "A2 convergence", etc.

AI-generated text was explicitly rejected for this role. The rule engine is synchronous — it produces the insight before the DOM is even built. An AI call would add 1–3 seconds of latency between the submission response arriving and the modal appearing, which would feel like a loading stall at the exact moment the learner is expecting immediate feedback.

Next-exercise hint

A single chip in the insight panel, prefixed with →, suggests what to do next. This is also rule-based:

  • Struggled (5+ attempts OR over the time benchmark) → "Consolidate: another [difficulty] [category] exercise."
  • Excelled (first-try AND under 40% of T_d) AND not already at advanced → "Strong run — try an [next difficulty] [category] challenge."
  • Otherwise → "Continue: next [difficulty] [category] exercise."

This hint is advisory only. The learner can ignore it and use any of the three footer buttons regardless of what it says.

Navigation buttons

ButtonStyleWhat happensWhy this position
Dashboard Ghost (secondary) Navigates to codivium_insights_embedded.html Listed first because it is the most likely destination for a learner who wants to see their updated mastery scores immediately after a completion.
Return to Menu Ghost (secondary) Navigates to result.returnMenuUrl — e.g. menu-page.html?track=micro Second because it continues the current practice session — the learner goes back to the list and picks another exercise.

Section 5

Data flow end to end

The following describes every data movement in one complete pass through the practice loop, from the learner opening the menu to the feedback modal appearing.

Step 1 — Menu page loads

Browser reads ?track=micro from URL. JS calls GET /api/menu/payload?track=micro. Server queries: all exercises for the micro track from the exercise catalogue, plus the user's coding_sessions table to determine completion status per exercise. Returns one JSON payload. JS renders all cards into the grid, populates filter checkboxes from payload.categories.

Step 2 — Filter / search / sort

All client-side. No server calls. The JS filter function reads the active checkboxes and search field text, then shows/hides and re-orders card DOM nodes. Instant response.

Step 3 — Learner clicks a card

The card's href navigates to editor.html?id=micro_042&from=menu-page.html%3Ftrack%3Dmicro. Normal browser navigation. The menu page is discarded.

Step 4 — Editor page loads

Browser reads ?id=micro_042 from URL. JS calls GET /api/exercise?id=micro_042. Server queries the exercises table for the problem content plus the user's attempt history for this exercise. Returns the exercise payload. Editor renders the problem statement, populates the code editor with the scaffold, shows the learner's prior history if any.

Step 5 — Learner submits code

JS POSTs to /api/exercise/submit with the code and session ID. Server runs the test suite. Two outcomes:

  • Failure: response accepted: false. Editor updates test results panel. Session data is written to coding_attempts for this submission. No modal.
  • Success: response accepted: true with full feedback object. Editor writes the session to coding_sessions (marking it accepted). Calls CodiviumFeedback.show(result.feedback).

Step 6 — Feedback modal appears

The modal builds itself from the feedback object in memory. No additional server calls. The insight text is generated synchronously by the rule engine. If feedback.history has 2+ entries, the chart SVGs are drawn 50ms after DOM insertion (to allow the container dimensions to be computed). The modal animates in. The editor and the learner's accepted code are visible behind it.

Step 7 — Learner chooses

  • Dashboard: navigates to codivium_insights_embedded.html. The dashboard's next load will pick up the new session data (cache TTL 60s). The Codivium Score and all metrics reflect the new evidence.
  • Return to Menu: navigates to result.returnMenuUrl. The menu page reloads from the API — the completed exercise now shows a green tick.
  • Try Again: navigates to editor.html?id=micro_042&from=menu-page.html%3Ftrack%3Dmicro. A fresh editor session starts. The prior attempt history is now one longer.

Code execution is infrastructure, not application code

The /api/exercise/submit endpoint needs to run the learner's code against the test suite in an isolated execution environment. This is a sandboxed runtime (e.g. a Docker container, a WASM runtime, a cloud code execution service). It is the most significant infrastructure piece in the entire system and needs to be designed independently of the application layer described in this document.

Section 7

Interface contracts summary

The precise data shapes that connect each piece of the system.

Menu payload (GET /api/menu/payload)

{
  track: string,           // "micro" | "interview"
  trackLabel: string,      // "Micro Challenges" | "Interview Preparation"
  categories: string[],    // 14 canonical category names
  difficultyLevels: string[], // ["basic","intermediate","advanced"]
  exercises: Array<{
    id: string,            // globally unique, e.g. "micro_042"
    name: string,
    shortDescription: string,
    category: string,      // one of the 14 categories
    difficulty: string,    // "basic" | "intermediate" | "advanced"
    completionStatus: string, // "completed" | "attempted" | "not_started"
    completedAt: string | null // ISO date of most recent acceptance
  }>
}

Exercise payload (GET /api/exercise)

{
  id, name, category, difficulty,
  problemStatement: string,
  constraints: string[],
  examples: Array<{ input, output, explanation? }>,
  codeScaffold: string,
  testsTotal: number,
  priorAttempts: number,
  bestPriorScore: number,    // % peak tests passed, 0-100
  lastSolvedAt: string | null
}

Feedback payload (nested in submission response)

{
  exerciseId, exerciseName, category, difficulty,
  testsPassed, testsTotal, isFirstSolve,
  returnMenuUrl: string,     // decoded from editor's ?from= param
  current: {
    timeToSuccessSeconds,
    attempts,
    bestPreSuccessPercent,
    timeToFirstAttemptSecs,
    avgIterationSeconds
  },
  history: Array<{ timeToSuccessSeconds, attempts }>, // oldest first, max 8
  deltas: {
    timeToSuccess: number | null,  // % change vs previous, null if first
    attempts: number | null
  }
}

Card data attributes (generated by menu page JS renderer)

<a class="exercise-card"
   data-id="{exercise.id}"
   data-cat="{exercise.category.toLowerCase()}"
   data-level="{exercise.difficulty}"
   data-status="{exercise.completionStatus}"
   data-name="{exercise.name.toLowerCase()}"
   data-desc="{exercise.shortDescription.toLowerCase()}"
   href="#" data-illustrative="true" title="URL template — not a live link">
